# AI 自动化测试平台设计总览

## 1. 背景

参考文章《公司要我搭AI测试平台，完全没经验怎么办？救命！》的建设思路，平台不应只做“大模型问答工具”，也不应直接追求“AI 全自动测试银弹”，而应定位为：

> 面向智能硬件、移动 App、嵌入式设备和项目制交付的“全栈测试管理平台 + AI 能力”。

文章中将 AI 测试平台分为三类：

| 类型 | 说明 | 本项目取舍 |
| --- | --- | --- |
| A 类：AI 辅助工具集成 | 接入大模型生成测试用例、测试点、报告说明 | 必须支持，但不能止步于此 |
| B 类：AI 驱动自动化平台 | AI 识别 UI、自动修复定位、自然语言驱动执行 | 分阶段支持，先接入可控执行能力，再增强 AI 自愈 |
| C 类：全栈测试管理平台 + AI 能力 | 需求、用例、执行、缺陷、报告、自动化、权限统一管理 | 本项目核心定位 |

## 2. 项目目标

建设一套模块化、可配置、可私有化部署的 AI 自动化测试框架，覆盖：

- 需求接入：支持本地导入、Jira、飞书项目等需求来源。
- AI 分析：支持本地或企业自有大模型配置，对需求进行测试点识别、风险分析、用例生成。
- 用例管理：用例分为自动化测试和手工测试两大类，支持 AI 用例采纳、启用、废弃、人工编辑、版本管理、评审流转。
- 自动化执行：支持接口、App UI、嵌入式设备、ADB Shell、串口日志分析、框架级环境初始化和自动日志采集等执行能力。
- 配置中心：支持权限、大模型、邮件、通知、命令安全、字段映射、工具参数等多层级配置。
- 工具箱/Skill：沉淀功耗采集、继电器控制、串口、ADB、日志分析等可复用能力，供用例和 AI 助手调用。
- AI 助手：提供显眼的自然语言入口，支持一句话生成用例、查询项目质量、触发用例集合执行并输出报告。
- 任务调度：支持手动执行、定时执行、每日/每周回归执行。
- 报告与解读：支持执行报告、版本/迭代报告、周期质量报告、AI 风险分析、趋势总结、通知推送。
- 协同与账号体系：支持账号创建、邀请加入、SSO/LDAP 扩展、团队/项目成员协同。
- 权限与项目管理：支持多项目、多角色、多租户隔离、项目级权限和项目级配置。
- 通知集成：支持企业微信/飞书机器人、邮件、Webhook 等可扩展通知方式。

## 3. 设计原则

- 平台化：需求、用例、执行、报告、缺陷、设备统一管理。
- 模块化：AI、执行器、通知、数据源、设备通道都采用插件化设计。
- 可配置：模型、Prompt、字段映射、执行环境、通知策略均可配置。
- 可审计：AI 生成内容、人工采纳、用例变更、执行记录、报告解读均可追踪。
- 可私有化：支持本地模型、内网部署、企业账号体系、敏感数据脱敏。
- 面向智能硬件：原生支持 ADB、串口、日志、设备池、固件版本、硬件批次。

## 4. MVP 闭环

第一阶段不追求一次性完成所有能力，优先打通以下主链路：

```text
账号初始化/成员邀请
  -> 系统/项目配置初始化
  -> 项目创建
  -> 需求导入/同步
  -> AI 分析需求
  -> 生成候选测试用例
  -> 人工采纳/编辑/启用
  -> 创建测试计划
  -> 选择设备和执行环境
  -> 手动、定时或 AI 助手自然语言触发执行
  -> 调用工具箱/执行器采集日志和结果
  -> 生成报告
  -> AI 解读报告
  -> 通知相关人员
```
