# PRD：AI 自动化测试平台

## 1. 产品定位

本产品是面向智能硬件、嵌入式设备、移动 App 与项目制交付团队的 AI 自动化测试协同平台，提供从账号协同、项目管理、需求接入、AI 用例生成、用例管理、自动化执行、日志分析、报告解读、权限管理到质量运营的一体化能力。

产品不是单纯的 AI 聊天工具，而是将 AI 能力嵌入测试生命周期：

- 在需求阶段辅助识别测试点、风险点和覆盖盲区。
- 在设计阶段生成结构化测试用例和自动化候选步骤。
- 在执行阶段调度设备、调用工具箱/Skill、执行脚本、收集日志和结果。
- 在分析阶段自动归因、总结失败原因和生成报告解读。
- 通过显眼的 AI 助手入口，让用户用自然语言生成用例、发起执行、查询报告和定位失败。

## 3. 核心业务流程

### 3.1 需求到用例

```text
需求来源配置 -> 需求导入/同步 -> 字段映射 -> AI 需求解析 -> 测试点生成 -> 候选用例生成 -> 人工评审 -> 采纳/废弃/启用 -> 用例入库
```

关键规则：

- AI 生成的用例默认为“候选”状态，不直接进入执行。
- 候选用例必须经过人工采纳后才能成为正式用例。
- 用例启用与采纳分离：采纳代表认可内容，启用代表可进入计划执行。
- 每次 AI 重新生成或人工修改都应形成用例版本记录。

### 3.2 用例到执行

```text
选择项目 -> 创建测试计划 -> 选择用例集 -> 绑定环境/设备 -> 选择执行器 -> 手动/定时触发 -> 执行采集 -> 结果归档 -> 报告生成
```

关键规则：

- 一个测试计划可包含多个测试套件。
- 一个执行任务可绑定一个或多个设备。
- App UI 测试可基于 uiautomator2 执行。
- 嵌入式测试可执行 ADB Shell、串口命令、日志匹配、关键字断言。
- 执行结果必须保留原始日志、截图、视频或关键证据。

### 3.3 AI 助手自然语言执行

```text
用户输入自然语言 -> AI 意图识别 -> 权限和项目上下文确认 -> 生成执行计划 -> 用户确认 -> 调用用例/工具箱/执行器 -> 汇总结果 -> 输出报告和 AI 解读
```

典型指令：

- “帮我给 WiFi 配网需求生成 P0/P1 测试用例。”
- “执行智能门锁项目 Android App 的冒烟用例集，使用实验室 1 号设备。”
- “跑一下固件 1.2.3 的功耗回归，并生成报告发到飞书群。”
- “分析今天失败最多的用例，按可能原因分组。”

关键规则：

- AI 助手只能在用户已有权限范围内查询和执行。
- 涉及真实设备、继电器、电源、刷机、重启、删除数据等高风险动作，必须二次确认。
- AI 生成的执行计划必须可审计，包括识别意图、选择的项目、用例集、设备、工具和参数。
- 自然语言执行不是绕过用例管理，而是复用用例库、测试计划和工具箱能力。

### 3.4 报告到通知

```text
执行结束 -> 汇总结果 -> AI 解读 -> 生成报告 -> 发送通知 -> 失败项跟踪 -> 缺陷联动
```

关键规则：

- 报告支持项目维度、计划维度、执行批次维度、日报、周报。
- AI 解读必须展示原始数据依据，避免只输出结论。
- 通知渠道应可配置，支持飞书、企业微信、邮件、Webhook。

执行配置继承规则：

```text
运行时手动参数 > 用例级执行配置 > 测试计划配置 > 项目执行配置 > 系统默认配置
```

循环 loop 策略：

| 配置项 | 说明 |
| --- | --- |
| loop_enabled | 是否启用循环执行 |
| loop_count | 循环次数，默认 1 |
| loop_interval_seconds | 每轮间隔 |
| stop_on_first_failure | 首次失败是否停止后续循环 |
| collect_logs_each_loop | 是否每轮单独采集日志 |
| report_each_loop | 是否展示每轮结果明细 |

中断策略：

- 用户可手动中断执行批次、测试计划或单个正在执行的用例。
- 中断后状态标记为 `interrupted` 或 `cancelled`，已完成结果保留，未执行用例标记为 `skipped`。
- 中断必须生成本地报告，包含已执行结果、已采集日志、中断时间、中断人和原因。
- 中断报告默认不发送飞书、企业微信、邮件等外部通知，除非项目通知策略显式开启。

执行器类型：

| 类型 | 能力 |
| --- | --- |
| ManualExecutor | 人工执行记录结果 |
| ApiExecutor | REST API 测试、断言、变量提取 |
| AppUiExecutor | 基于 uiautomator2 的 Android App UI 测试 |
| AdbExecutor | ADB Shell 命令、设备信息、安装卸载、日志采集 |
| SerialExecutor | 串口命令、串口日志分析、关键字匹配 |
| ScriptExecutor | Python/Shell 自定义脚本 |
| AiAgentExecutor | 后续支持自然语言任务驱动执行 |

## 6. 验收标准

MVP 验收必须满足：

- 能创建项目和用户角色。
- 能配置至少一个 OpenAI-compatible 大模型。
- 能导入一条需求并生成候选测试用例。

- 能创建测试计划并手动执行用例。

- 能通过 uiautomator2 执行一个 Android UI 操作示例。
- 能生成执行报告并通过至少一种通知方式发送。
- 能生成 AI 报告解读。

| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| HOOK-001 | 支持执行生命周期 Hook：pre_run、pre_case、post_case、post_run | P0 |
| HOOK-002 | 支持在 pre_run/pre_case 自动初始化 Android 设备和外接设备 | P0 |
| HOOK-003 | 支持在 post_case 自动采集日志、截图、串口日志、logcat 和工具输出 | P0 |

| HOOK-006 | 支持自动化用例无需显式编写公共日志采集逻辑 | P0 |

推荐 Hook 流程：

```text
pre_run：准备执行环境、拉取代码、检查设备、初始化全局资源
pre_case：初始化单用例环境、ADB 恢复 App/设备状态、开始串口/logcat 采集
case_run：执行 Python 自动化用例
post_case：停止日志采集、拉取失败日志、上传云端、生成附件链接
post_run：汇总报告、释放资源、生成趋势和 AI 解读
```

Android 初始化动作建议：

- `adb wait-for-device` 等待设备在线。
- `adb shell input keyevent KEYCODE_HOME` 回到桌面。
- `adb shell am force-stop <package>` 停止目标 App。
- `adb shell pm clear <package>` 清理 App 数据，可配置是否执行。
- 安装或覆盖安装指定 APK。
- 清理 logcat 缓冲区。
- 设置屏幕常亮、解锁、关闭干扰弹窗。
- 采集设备属性、系统版本、App 版本作为环境快照。
