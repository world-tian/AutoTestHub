# 技术架构设计

## 1. 架构目标

平台需要同时满足测试管理、AI 生成、自动化执行、硬件设备接入、报告分析、企业集成、统一配置、工具箱复用和自然语言操作能力。建议采用“前后端分离 + 异步任务队列 + 插件化执行器 + 可配置集成中心 + 工具箱/Skill 注册中心 + 可控 AI 助手”的架构。

## 2. 推荐技术栈

| 层级 | 推荐选型 | 原因 |
| --- | --- | --- |
| 前端 | Vue 3 + TypeScript + Vite + Element Plus/Naive UI | 管理后台生态成熟，开发效率高 |
| 后端 | Python 3.12 + FastAPI | 与 AI、uiautomator2、串口、ADB、自动化生态匹配度高 |
| ORM | SQLAlchemy 2.x + Alembic | 类型清晰，迁移成熟 |
| 数据库 | PostgreSQL 16 | JSONB、全文检索、复杂查询、扩展能力优于 MySQL |
| 缓存 | Redis | 会话、任务锁、设备锁、队列状态 |
| 任务队列 | Celery/RQ + Redis，后续可切换 RabbitMQ | 执行任务、AI 任务、报告任务异步化 |
| 文件存储 | MinIO/S3、本地文件可选 | 保存日志、截图、报告、附件 |
| AI 接入 | OpenAI-compatible Client + Provider 插件 | 兼容本地模型、DeepSeek、通义、智谱等 |
| AI 助手 | LangGraph/自研 Planner + Tool Calling | 支持自然语言生成用例、查询和可控执行 |
| 工具箱 | Python Plugin + HTTP Tool + Shell Tool | 封装功耗、继电器、日志分析等可复用能力 |
| 配置中心 | 数据库配置 + 加密密钥 + 版本审计 | 支持权限、模型、邮件、通知、工具等大量配置项 |
| App UI | uiautomator2 | Android App 自动化成熟，适合智能硬件配套 App |
| ADB | adbutils / subprocess adb | 设备发现、shell、logcat、安装卸载 |
| 串口 | pyserial | 串口命令与日志采集标准方案 |
| 报告 | HTML + ECharts + 可选 WeasyPrint/PDFKit | 可视化和导出能力 |
| 权限 | JWT + RBAC + 项目级 ACL | 易于前后端分离和 API 权限控制 |
| 部署 | Docker Compose，后续 Kubernetes | 先快速落地，再支持规模化 |

说明：文章中提到 TestHub 使用 Django + Vue + MySQL。若团队已有 Django 经验，也可选择 Django Rest Framework。考虑本项目有较强自动化执行、设备控制和 AI 编排需求，FastAPI 更轻量，更适合作为可插拔执行平台。

## 4. 服务拆分建议

MVP 阶段可以采用模块化单体，避免过早微服务化：

```text
backend/
  app/
    api/              # REST API
    core/             # 配置、鉴权、权限、异常
    modules/
      auth/
      project/
      requirement/
      testcase/
      review/
      execution/
      device/
      report/
      integration/
      configuration/
      toolbox/
      assistant/
      ai/
      notification/
    workers/          # 异步任务
    executors/        # 执行器插件
    providers/        # 模型、数据源、通知 Provider
```

当执行规模扩大后，可拆分为：

| 服务 | 职责 |
| --- | --- |
| api-service | Web API、权限、项目、用例、配置 |
| ai-service | 模型调用、Prompt 编排、RAG、AI 任务 |
| execution-service | 任务调度、执行队列、结果回写 |
| device-agent | 部署在测试机，控制 ADB、串口、App UI |
| report-service | 报告生成、AI 解读、通知发送 |

## 5. 设备 Agent 架构

智能硬件测试建议引入 Agent，部署在连接设备的 Mac/Linux/Windows 测试机上。

```text
平台服务
  -> 下发执行任务
  -> Agent 拉取任务
  -> Agent 锁定设备
  -> 执行 ADB/uiautomator2/串口/脚本
  -> 上传日志、截图、结果
  -> 平台生成报告
```

Agent 能力：

- 注册测试机和本机能力。
- 发现 Android 设备和串口设备。
- 执行任务时进行设备锁定。
- 上传执行日志、截图、串口日志和附件。
- 支持离线心跳检测和任务超时回收。

## 6. AI 架构

AI 能力应抽象为 Provider + Prompt Template + Task Pipeline。

```text
用户输入/需求内容
  -> 数据清洗与脱敏
  -> Prompt 模板渲染
  -> 模型 Provider 调用
  -> 结构化输出解析 JSON Schema
  -> 质量校验
  -> 人工确认
  -> 入库
```

Provider 类型：

| Provider | 示例 |
| --- | --- |
| openai_compatible | vLLM、Ollama、LM Studio、DeepSeek、通义、智谱 |
| local_http | 企业内部模型服务 |
| mock | 测试环境固定返回 |

AI 结果必须结构化保存：

- 输入来源和摘要。
- Prompt 模板版本。
- 模型名称和参数。
- 输出 JSON。
- 人工采纳状态。
- 失败原因和重试记录。

## 7. 执行器插件接口

执行器统一接口：

```python
class BaseExecutor:
    type: str

    def validate(self, case, environment):
        pass

    def prepare(self, context):
        pass

    def execute(self, context):
        pass

    def collect_artifacts(self, context):
        pass

    def teardown(self, context):
        pass
```

执行结果统一模型：

| 字段 | 说明 |
| --- | --- |
| status | passed/failed/blocked/skipped/error |
| started_at | 开始时间 |
| ended_at | 结束时间 |
| duration_ms | 耗时 |
| logs | 执行日志摘要 |
| artifacts | 附件列表 |
| assertions | 断言结果 |
| error_message | 错误信息 |

## 8. 安全设计

- 模型 API Key、Jira Token、飞书 Secret 必须加密存储。
- 项目数据按 project_id 隔离，接口层强制校验项目权限。
- AI 调用前支持字段脱敏，如账号、手机号、SN、客户名称。
- ADB/串口命令支持白名单/黑名单，防止高危命令误执行。
- 操作审计记录用户、动作、对象、前后值摘要和 IP。

## 11. 配置中心架构

配置中心负责统一管理平台所有可变参数，避免配置散落在代码、环境变量和各模块中。

配置分层：

```text
SystemConfig -> ProjectConfig -> PlanConfig -> CaseConfig -> RuntimeOverride
```

配置类别：

| 类别 | 示例 |
| --- | --- |
| 权限配置 | 角色、权限点、审批规则、项目成员策略 |
| 模型配置 | base_url、model_name、temperature、api_key、场景绑定 |
| 邮件配置 | SMTP、发件人、模板、收件组 |
| 通知配置 | 飞书、企业微信、Webhook、事件订阅 |
| 需求配置 | Jira JQL、飞书字段映射、同步周期 |
| 执行配置 | 并发数、超时、重试、Agent 选择策略 |
| 安全配置 | 命令黑白名单、危险动作确认、脱敏规则 |
| 工具配置 | 功耗仪、继电器、串口服务器、实验室资源 |

关键设计：

- 敏感值进入 `secrets` 表或外部 Secret Manager。
- 普通配置进入 `configuration_items`，支持 JSON Schema 校验。
- 配置变更进入 `configuration_versions`，支持审计和回滚。
- 运行时读取配置时生成解析快照，确保执行可复现。

## 12. 工具箱/Skill 架构

工具箱是可复用能力注册中心，统一封装功耗采集、继电器控制、日志分析、ADB 辅助能力、串口辅助能力、数据生成等工具。

```text
Tool Definition -> Tool Instance -> Tool Permission -> Tool Invocation -> Tool Result/Artifact
```

工具接入形态：

| 形态 | 说明 |
| --- | --- |
| Python Plugin | 平台内置或本地插件，适合功耗仪 SDK、继电器 SDK |
| HTTP Tool | 调用实验室已有服务，如功耗平台、设备控制平台 |
| Shell Tool | 调用命令行工具，需受黑白名单和沙箱限制 |
| Agent Tool | 由 Device Agent 在测试机执行，适合本地 USB/串口设备 |

工具调用方：

- 测试用例步骤。
- 测试计划前置/后置动作。
- AI 助手 Tool Calling。
- 报告分析任务。
- 人工工具调试页面。

## 13. AI 助手架构

AI 助手采用可控 Agent，而不是完全自主执行。

```text
Chat UI
  -> Assistant API
  -> Intent Router
  -> Context Retriever
  -> Planner
  -> Permission Guard
  -> User Confirmation
  -> Tool/Test Execution
  -> Report/Answer Generator
```

意图类型：

| 意图 | 示例 |
| --- | --- |
| generate_cases | 根据需求生成测试用例 |
| query_status | 查询项目、计划、执行、报告状态 |
| run_cases | 执行某项目某用例集合 |
| run_tool | 调用功耗、继电器、日志分析等工具 |
| analyze_failure | 分析失败日志和报告 |
| create_report | 生成日报、周报或专项报告 |

安全护栏：

- 所有动作先做 RBAC 和项目 ACL 校验。
- 高风险工具必须二次确认。
- AI 不能越权读取其他项目数据。
- AI 执行计划、工具入参和结果必须落库审计。
- AI 只能调用工具注册中心暴露的工具，不能任意执行系统命令。

## 8. AutoTestRunner (本地执行引擎) 深度设计

为了满足开源社区的本地使用诉求，并支持复杂的智能硬件测试，`AutoTestRunner` 绝对不仅仅是一个简单的数据上报插件，而是一个**重度封装的本地自动化测试框架**。

它可以完全脱离云端独立运行，也可以与云端联动。

### 8.1 核心能力矩阵
AutoTestRunner 在本地提供以下完整能力：
1. **测试循环控制 (Loop Controller)**：支持本地配置用例循环次数（如压力测试跑 1000 次）、失败重试、中断控制。
2. **硬件工具链集成 (Toolchains)**：
   - **ADB 模块**：封装常用的设备连接、状态恢复、Logcat 抓取、App 安装清理。
   - **串口模块**：后台守护进程抓取串口日志，支持正则匹配错误堆栈。
   - **外设控制**：内置针对常用功耗仪、继电器（断电重启）的控制 SDK。
3. **安全执行引擎 (Try-Except-Finally)**：严格遵循生命周期 Hook，保障异常发生时依然能安全释放本地硬件锁、保存错误现场。
4. **本地报告与通知 (Local Reporter & Notifier)**：
   - 即使没有云端，本地执行完毕后也可自动生成本地 HTML 测试报告。
   - 支持在本地直接读取配置，将测试结果推送到飞书/企业微信/邮件，无需经过云端中转。

### 8.2 云端与本地职责边界与交互协议

为了让平台好用且符合工程直觉，我们严格划定了“云端管控”与“本地执行”的职责边界：

#### ☁️ 云端管控平台 (AutoTestHub)
云端是**资产管理、协同与数据大盘**的中心，主要负责：
- **项目与需求管理**：导入需求、规划版本/迭代。
- **用例大盘管理**：统筹所有测试用例。手工用例直接在云端编写与执行；自动化用例在云端表现为**只读视图**（从本地同步而来）。
- **人工测试执行**：手工测试用例的步骤流转、结果录入、缺陷关联。
- **全局配置中心**：维护全局环境变量、通知策略、模型配置。
- **质量度量与 AI 分析**：基于收集上来的数据，进行 AI 失败聚类、项目周期风险评估。

#### 💻 本地执行框架 (AutoTestRunner)
本地是**代码编写、硬件驱动与自动化执行**的中心，主要负责：
- **自动化代码编写**：提供 Python SDK，开发者在本地 IDE 编写 `pytest` 脚本。
- **本地 AI 辅助**：通过 CLI 工具或 IDE 插件，利用 AI 本地生成测试脚本骨架。
- **测试执行与资源调度**：控制本地硬件（串口、ADB、继电器），执行 `try-except-finally` 安全循环。
- **结果与日志收集**：在本地抓取设备的原始 Logcat、串口 Dump 及报错截图。
- **本地报告发送**：直接在本地生成 Allure/HTML 报告，并通过 Webhook 发送企微/飞书通知。

#### 🌉 云端与本地如何打通？(交互协议)
1. **代码元数据同步 (Sync Up)**：
   本地代码通过专属装饰器打标，例如 `@autotest(req_id="REQ-123", title="蓝牙连接")`。
   开发者在本地执行 `autotest-runner sync`，框架会通过 AST 解析这些装饰器，将自动化用例的标题、所属需求等元数据通过 OpenAPI 同步到云端平台。
2. **执行结果上报 (Report Up)**：
   本地执行 `autotest-runner run --report-to-cloud` 时，执行完毕后会自动调用云端 API，将执行状态（Passed/Failed）、日志附件链接、错误摘要上报，并在云端生成一条“执行批次”记录。
3. **云端下发调度 (Dispatch Down - 进阶)**：
   本地 Runner 可以开启 Daemon 模式轮询云端。云端发起“自动化测试计划”时，将任务指令（带上全局环境变量）下发给闲置的本地 Runner 进行物理执行。

这种设计让 AutoTestRunner 自身具备极高的开源传播价值。
