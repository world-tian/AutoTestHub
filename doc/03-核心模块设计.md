# 核心模块详细设计

## 1. 模块清单

| 模块 | 核心职责 |
| --- | --- |
| 用户权限模块 | 账号创建、邀请、登录、SSO 扩展、RBAC、项目权限、审计日志 |
| 团队协同模块 | 用户组、项目成员、待办、评论、通知、评审协作 |
| 项目管理模块 | 项目、成员、项目角色、权限、模块、版本、环境、配置 |
| 需求管理模块 | 需求导入、同步、字段映射、版本管理 |
| AI 分析模块 | 需求分析、用例生成、报告解读、Prompt 管理 |
| 用例管理模块 | 候选用例、正式用例、启用、版本、标签、导入导出 |
| 评审模块 | 用例评审、评论、审批、评审记录 |
| 执行模块 | 测试计划、测试套件、执行任务、调度、结果 |
| 设备模块 | Android 设备、串口设备、设备锁、Agent |
| 报告模块 | 执行报告、日报、周报、趋势、AI 解读 |
| 集成模块 | Jira、飞书、通知机器人、邮件、Webhook |
| 缺陷模块 | 缺陷创建、关联、同步、AI 归因 |
| 系统配置模块 | 权限、大模型、Prompt、邮件、通知、命令安全、全局参数、配置版本 |
| 工具箱/Skill 模块 | 功耗采集、继电器控制、日志分析、数据生成、通用硬件能力封装 |
| AI 助手模块 | 自然语言生成用例、查询状态、触发执行、调用工具、输出报告 |

## 2. 用户权限模块

### 2.1 权限模型

采用 RBAC + 项目 ACL：

```text
User -> Role -> Permission
User -> Team -> TeamMember
User/Team -> ProjectMember -> ProjectRole -> ProjectPermission
```

权限粒度建议：

| 资源 | 动作 |
| --- | --- |
| project | create/read/update/delete/archive/manage_members/manage_roles/manage_config |
| requirement | import/sync/read/update/delete/analyze |
| testcase | create/read/update/delete/adopt/enable/review/export |
| execution | create/run/stop/retry/read/delete |
| device | create/read/update/delete/lock/unlock |
| report | create/read/export/notify/ai_analyze |
| integration | create/read/update/delete/test_connection |
| system | manage_users/manage_models/manage_audit/manage_config |
| toolbox | create/read/update/delete/invoke/debug |
| assistant | chat/generate_cases/run_cases/run_tools |

### 2.2 审计事件

必须记录：

- 登录成功/失败。
- 创建、修改、删除项目配置。
- 新增或修改模型密钥。
- AI 生成用例和人工采纳。
- 用例版本变更。
- 执行任务触发、停止、重跑。
- 报告发送。

## 3. 项目管理模块

### 3.1 项目维度

项目是所有业务数据的主隔离边界。

项目下包含：

- 项目成员。
- 需求来源配置。
- 模型配置和 Prompt 模板。
- 用例库。
- 测试计划。
- 设备池绑定。
- 报告和通知策略。

### 3.2 项目配置

| 配置项 | 说明 |
| --- | --- |
| 默认模型 | AI 需求分析、用例生成、报告解读使用的模型 |
| 默认 Prompt | 各 AI 场景的默认 Prompt 模板 |
| 用例字段模板 | 不同项目可自定义用例字段 |
| 通知策略 | 计划完成、失败、日报、周报通知渠道 |
| 需求字段映射 | Jira/飞书字段到平台字段映射 |
| 设备池 | 项目可使用的设备范围 |

## 4. 需求管理模块

### 4.1 导入方式

| 方式 | 说明 |
| --- | --- |
| 手工录入 | 标题、描述、验收标准、附件 |
| 文档导入 | Markdown、Word、Excel、PDF |
| Jira 同步 | JQL 查询、字段映射、定时同步 |
| 飞书同步 | 飞书项目或多维表格接口同步 |
| API 导入 | 提供开放 API 给其他系统推送 |

### 4.2 需求状态

```text
imported -> parsed -> ai_analyzed -> cases_generated -> covered -> archived
```

### 4.3 AI 分析输出

AI 分析需求后生成：

- 需求摘要。
- 功能点列表。
- 测试点列表。
- 风险点列表。
- 依赖条件。
- 适合自动化的场景。
- 建议测试类型。

## 5. AI 分析模块

### 5.1 AI 场景

| 场景 | 输入 | 输出 |
| --- | --- | --- |
| 需求摘要 | 需求标题、描述、附件文本 | 摘要、范围、非范围 |
| 测试点识别 | 需求内容、历史缺陷 | 测试点、风险点 |
| 用例生成 | 测试点、用例模板 | 结构化用例 JSON |
| 用例补全 | 人工用例草稿 | 前置条件、步骤、预期补全 |
| 用例去重 | 候选用例列表 | 重复用例分组 |
| 报告解读 | 执行结果、失败日志 | 总结、风险、建议动作 |
| 失败归因 | 日志、截图、历史失败 | 可能原因、排查建议 |

### 5.2 Prompt 模板变量

```text
{{project_name}}
{{product_type}}
{{requirement_title}}
{{requirement_description}}
{{acceptance_criteria}}
{{hardware_model}}
{{firmware_version}}
{{app_version}}
{{case_template}}
{{historical_defects}}
{{risk_preferences}}
```

### 5.3 输出质量控制

- 使用 JSON Schema 校验结构化输出。
- 对缺失字段进行自动重试或提示人工补全。
- 对高风险场景要求人工确认。
- 支持人工反馈“有用/无用/需修改”。
- 记录每次 AI 结果与最终采纳结果的差异。

## 6. 用例管理模块

### 6.1 用例分类

用例先分两大类，再细分执行类型：

| 大类 | 说明 |
| --- | --- |
| manual | 手工测试用例，由测试人员在平台上逐步执行和记录结果 |
| automated | 自动化测试用例，统一使用 Python 编写，通过执行器自动运行 |

自动化执行类型：

| 类型 | 说明 |
| --- | --- |
| api | Python 接口自动化用例 |
| app_ui | Python + uiautomator2 Android App UI 自动化用例 |
| adb | Python 封装 ADB Shell/Logcat 用例 |
| serial | Python + pyserial 串口命令/串口日志分析用例 |
| toolbox | Python 调用工具箱，如功耗、继电器、日志分析 |
| mixed | Python 编排多步骤混合用例 |

### 6.2 用例状态

| 状态 | 说明 |
| --- | --- |
| ai_candidate | AI 生成候选用例 |
| adopted | 已采纳但未评审 |
| reviewing | 评审中 |
| approved | 评审通过 |
| enabled | 已启用，可加入计划 |
| disabled | 停用 |
| rejected | 已拒绝 |
| deprecated | 已废弃 |

### 6.3 用例版本

触发新版本的动作：

- AI 重新生成。
- 人工修改标题、步骤、预期、前置条件。
- 绑定自动化脚本或修改执行配置。
- 评审通过后固化基线。

版本字段：

- version_no：递增版本号。
- change_type：ai_generate/manual_edit/review_update/script_bind。
- change_summary：变更摘要。
- content_snapshot：完整快照。
- created_by：变更人或 AI 任务。

## 7. 评审模块

### 7.1 评审对象

- 单条用例。
- 用例集。
- 某个需求生成的全部候选用例。
- 某个版本迭代的新增/变更用例。

### 7.2 评审流程

```text
发起评审 -> 指定评审人 -> 评论/修改建议 -> 通过/驳回 -> 状态流转 -> 记录归档
```

### 7.3 评审规则

- P0 用例必须评审通过才能启用。
- AI 生成用例批量采纳后可批量发起评审。
- 评审驳回需要填写原因。
- 评审通过后生成新的用例版本基线。

## 8. 执行模块

### 8.1 核心对象

| 对象 | 说明 |
| --- | --- |
| 测试计划 | 一次迭代、一次回归或一次专项测试的计划 |
| 测试套件 | 用例集合，可按模块/类型/标签组织 |
| 执行任务 | 一次具体执行，可手动或定时触发 |
| 执行步骤结果 | 每个步骤或断言的执行结果 |
| 执行附件 | 日志、截图、视频、串口日志、报告文件 |

### 8.2 调度策略

- 支持立即执行。
- 支持 Cron 表达式。
- 支持每日/每周固定时间执行。
- 支持按设备标签筛选设备。
- 支持失败重试次数配置。
- 支持执行超时和任务取消。

### 8.3 设备锁

执行前必须获取设备锁：

```text
检查设备在线 -> 检查项目权限 -> 获取锁 -> 执行任务 -> 上传结果 -> 释放锁
```

异常场景：

- Agent 离线时释放超时锁。
- 任务取消时释放锁。
- 设备断开时标记任务 error。

## 9. ADB 执行设计

### 9.1 支持动作

| 动作 | 示例 |
| --- | --- |
| shell | `adb shell getprop ro.build.version.release` |
| install | 安装 APK |
| uninstall | 卸载 App |
| push/pull | 推送或拉取文件 |
| logcat | 采集系统日志 |
| screencap | 截图 |
| reboot | 重启设备，默认需高权限确认 |

### 9.2 命令安全

- 默认禁用高危命令，如格式化、删除系统目录、刷机命令。
- 项目可配置命令白名单。
- 执行命令必须记录执行人、设备、时间和输出。

## 10. 串口执行设计

### 10.1 串口配置

| 参数 | 说明 |
| --- | --- |
| port | 串口路径，如 `/dev/tty.usbserial-xxx` |
| baudrate | 波特率，如 115200 |
| bytesize | 数据位 |
| parity | 校验位 |
| stopbits | 停止位 |
| timeout | 读取超时 |

### 10.2 分析能力

- 发送命令并等待响应。
- 按关键字判断成功/失败。
- 按正则提取版本号、错误码、状态字段。
- 按时间窗口保存串口日志。
- AI 解读异常日志。

```json

  "summary": "本次执行总体结论",

  "failure_clusters": [
    {"name": "登录失败", "cases": 5, "possible_reason": "接口超时"}

}
```

## 15. 工具箱/Skill 模块

### 15.1 模块定位

工具箱用于沉淀可复用测试能力，避免每个用例、每个项目重复造轮子。工具既可以被测试用例调用，也可以被 AI 助手调用。

### 15.2 工具类型

| 工具 | 能力 |
| --- | --- |
| PowerMeterTool | 开始/停止功耗采集，计算平均功耗、峰值、曲线，导出 CSV |
| RelayTool | 控制继电器通道开关，支持断电重启、模拟按键、上下电循环 |
| LogAnalyzerTool | 关键字搜索、正则提取、错误聚类、日志摘要 |
| AdbUtilityTool | 设备属性、截图、logcat、应用安装卸载的复用封装 |
| SerialUtilityTool | 串口命令、等待响应、正则提取、日志窗口采集 |
| DataFactoryTool | 生成手机号、账号、随机设备名、测试数据 |
| NetworkTool | Ping、端口探测、网络切换、弱网配置扩展 |
| FixtureTool | 夹具控制、按键机械臂、传感器模拟扩展 |

### 15.3 工具调用协议

```json
{
  "tool_name": "PowerMeterTool",
  "action": "measure",
  "params": {
    "duration_seconds": 60,
    "sample_rate": 10,
    "channel": "CH1"
  },
  "assertions": [
    {"type": "less_than", "field": "avg_current_ma", "value": 120}
  ]
}
```

### 15.4 用例中引用工具

```json
{
  "type": "tool",
  "tool_name": "RelayTool",
  "action": "power_cycle",
  "params": {"channel": 1, "off_seconds": 5},
  "expected": "设备重新上线"
}
```

### 15.5 工具安全

- 工具需要声明风险等级：low/medium/high/critical。
- high/critical 工具需要二次确认或审批。
- 工具实例按项目授权，避免误操作其他实验室资源。
- 每次调用保存输入、输出、执行人、设备、附件和耗时。

## 16. AI 助手模块

### 16.1 产品入口

AI 助手应作为一级入口出现在：

- 首页顶部或右下角悬浮入口。
- 项目概览页。
- 需求详情页。
- 用例库页面。
- 执行中心和报告详情页。

### 16.2 支持能力

| 能力 | 示例 |
| --- | --- |
| 一句话生成用例 | “根据这个需求生成智能门锁低电量告警测试用例” |
| 一句话执行 | “执行 A 项目的冒烟用例，使用 Android 设备 lab-phone-01” |
| 一句话工具调用 | “采集设备 X 待机 5 分钟功耗，并判断平均电流是否小于 100mA” |
| 一句话报告 | “生成本周 WiFi 模块回归报告并发到飞书群” |
| 对话式分析 | “为什么今天失败率升高？按模块和设备型号分析” |

### 16.3 执行控制

AI 助手发起执行必须经过以下步骤：

```text
解析意图 -> 补全项目/用例/设备/环境 -> 权限校验 -> 生成计划 -> 用户确认 -> 创建 execution_run -> 执行 -> 总结
```

### 16.4 计划确认示例

```json
{
  "intent": "run_cases",
  "project": "智能门锁",
  "suite": "Android App 冒烟用例",
  "device": "lab-phone-01",
  "tools": ["AdbUtilityTool", "LogAnalyzerTool"],
  "risk_level": "medium",
  "need_confirmation": true
}
```

### 16.5 边界

- AI 助手不直接绕过评审启用未启用用例。
- AI 助手不直接执行未注册工具。
- AI 助手不直接执行任意 Shell 命令。
- AI 助手对高风险硬件动作必须要求用户确认。

## 17. 账号与项目协同模块

### 17.1 账号来源

| 方式 | 说明 | 建议 |
| --- | --- | --- |
| 管理员创建 | 系统管理员在后台创建账号，分配初始角色 | MVP 必做 |
| 邀请加入 | 管理员或项目 Owner 通过邮箱/链接邀请用户加入平台或项目 | MVP 必做 |
| 自注册 | 用户自行注册，需管理员审核或限制企业邮箱域名 | 默认关闭 |
| SSO/OIDC | 企业统一登录，平台只维护项目角色和业务权限 | 二期支持 |
| LDAP/飞书组织同步 | 同步企业组织和用户，减少人工维护 | 二期支持 |

### 17.2 账号生命周期

```text
invited -> active -> disabled -> locked -> archived
```

规则：

- `invited` 用户未激活前不能登录。
- `active` 用户可登录，具体项目权限由项目成员关系决定。
- `disabled` 用户不能登录，但历史执行、评审、审计记录保留。
- `locked` 用户因连续登录失败或安全策略锁定。
- 用户离职或停用时，不物理删除，避免历史记录丢失。

| TestDeveloper | 自动化脚本、执行器、工具箱、设备 Agent 配置 |
| Developer | 查看需求、用例、失败日志、报告，参与缺陷分析 |
| ProductManager | 查看需求覆盖、用例评审、报告和风险 |
| Viewer | 只读查看授权范围内的数据 |

## 18. 执行循环、中断与日志管理

### 18.1 执行配置继承

执行配置采用分层继承，单个用例可以覆盖上层配置；如果用例没有特殊配置，则复用计划、项目或系统默认配置。

```text
RuntimeOverride > CaseExecutionConfig > PlanExecutionConfig > ProjectExecutionConfig > SystemExecutionConfig
```

有效配置在执行开始时解析并生成快照，保存到执行批次和用例结果中，避免后续配置变更影响历史报告。

### 18.2 loop 配置

| 配置项 | 说明 | 默认值 |
| --- | --- | --- |
| loop_enabled | 是否启用循环 | false |
| loop_count | 循环次数 | 1 |
| loop_interval_seconds | 每轮间隔 | 0 |
| stop_on_first_failure | 首次失败是否停止 | false |
| continue_on_error | 某轮 error 后是否继续 | false |
| collect_logs_each_loop | 是否每轮独立采集日志 | true |
| max_duration_seconds | 最大总执行时长 | null |

单用例配置示例：

```json
{
  "execution_config": {
    "loop_enabled": true,
    "loop_count": 100,
    "loop_interval_seconds": 5,
    "stop_on_first_failure": true,
    "collect_logs_each_loop": true
  }
}
```

### 18.3 loop 执行结果

loop 用例结果应包含：

- 总循环次数。
- 已完成次数。
- 通过次数。
- 失败次数。
- 中断发生的轮次。
- 每轮开始/结束时间、状态、耗时、错误信息和日志引用。

### 18.4 中断处理

支持的中断方式：

| 类型 | 说明 |
| --- | --- |
| user_stop | 用户主动停止 |
| timeout | 达到最大执行时长 |
| device_lost | 设备断开 |
| agent_offline | Agent 离线 |
| safety_guard | 命中安全策略 |
| dependency_failed | 前置条件失败 |

中断规则：

- 已完成用例结果保留。
- 正在执行的用例标记为 `interrupted`。
- 未开始用例标记为 `skipped`。
- 释放设备锁和工具锁。
- 采集当前可用日志和附件。
- 生成本地中断报告。
- 默认不发送外部报告通知。

### 18.5 中断本地报告

中断本地报告内容：

| 内容 | 说明 |
| --- | --- |
| 基础信息 | 项目、计划、执行人、触发方式、开始/中断时间 |
| 中断信息 | 中断类型、中断人、原因、发生轮次或用例 |
| 已执行结果 | 已完成用例、每轮结果、通过/失败/错误统计 |
| 未执行范围 | 未执行用例、跳过原因 |
| 日志附件 | 执行日志、设备日志、串口日志、logcat、截图 |
| AI 解读 | 可选，默认不强制调用 AI，避免中断场景额外消耗 |

### 18.6 日志管理

日志类型：

| 日志 | 来源 |
| --- | --- |
| run_log | 执行批次日志 |
| case_log | 单用例日志 |
| step_log | 步骤日志 |
| loop_log | 循环轮次日志 |
| agent_log | Device Agent 日志 |
| adb_logcat | Android logcat |
| serial_log | 串口日志 |
| tool_log | 工具箱调用日志 |
| system_log | 平台服务日志引用 |

日志能力：

- 支持按项目、计划、执行批次、用例、设备、日志级别检索。
- 支持日志级别：debug/info/warn/error/fatal。
- 支持日志分片上传，避免大文件阻塞执行。
- 支持日志脱敏和敏感关键字遮蔽。
- 支持日志保留策略，按项目配置保留天数。
- 支持下载原始日志和查看结构化摘要。
- 支持 AI 对失败片段进行解读，但必须保留原始日志证据。

## 19. Python 用例与手工执行体验

### 19.1 Python 自动化用例规范

自动化用例统一使用 Python，推荐 pytest 作为默认执行框架。

推荐目录结构：

```text
automation/
  projects/<project_key>/
    cases/
      test_xxx.py
    pages/
    fixtures/
    tools/
    requirements.txt
```

绑定关系：

```text
平台用例 -> Git 仓库 -> 分支/tag -> Python 文件 -> 测试类/函数 -> pytest marker
```

自动化用例绑定信息：

| 字段 | 说明 |
| --- | --- |
| repo_url | Git 仓库地址 |
| branch | 分支或 tag |
| file_path | Python 文件路径 |
| module_name | Python 模块名 |
| class_name | 测试类名，可为空 |
| function_name | 测试函数名 |
| pytest_markers | pytest marker |
| runtime | Python 版本和运行环境 |
| dependencies | requirements.txt 或 poetry/uv 配置 |

### 19.2 AI 生成用例时的自动化判断

AI 生成用例必须输出：

- `case_category`：manual 或 automated。
- `automation_supported`：是否适合自动化。
- `automation_reason`：适合或不适合自动化的原因。
- `automation_type`：api/app_ui/adb/serial/toolbox/mixed。
- `automation_script_suggestion`：建议的 Python 脚本路径、函数名和实现思路。
- `manual_execution_tips`：如果是手工用例，给出执行注意事项和证据采集建议。

### 19.3 手工执行体验

手工执行用例集需要降低记录成本，重点优化失败记录、证据上传和缺陷联动。

手工执行页面建议：

| 区域 | 能力 |
| --- | --- |
| 用例导航 | 上一条/下一条、按模块筛选、只看未执行/失败 |
| 步骤执行 | 每步通过/失败/阻塞、填写实际结果、快捷备注 |
| 证据上传 | 拖拽上传、粘贴截图、手机扫码上传、批量上传 |
| 环境快照 | 自动记录设备、App、固件、网络、账号、执行人 |
| 失败复现 | 一键从步骤和实际结果生成复现步骤 |
| 缺陷联动 | 创建或关联 Jira/飞书项目/平台缺陷 |
| 草稿保存 | 中途保存，后续继续执行 |

### 19.4 图片和附件上传

支持方式：

- 拖拽图片/视频/日志到步骤或用例结果。
- 剪贴板粘贴截图。
- 移动端扫码上传到当前执行用例。
- App UI/ADB 自动截图作为附件。
- 串口、logcat、工具箱日志自动归档。

附件必须记录：

- 所属执行批次、用例、步骤。
- 上传人或采集来源。
- 文件类型、大小、hash、存储地址。
- 是否用于缺陷证据。

### 19.5 缺陷链接

失败或阻塞结果支持：

- 链接已有 Jira Issue。
- 链接已有飞书项目工作项。
- 创建平台内部缺陷。
- 从平台缺陷同步创建 Jira/飞书缺陷。
- AI 根据实际结果、复现步骤、附件和日志生成缺陷草稿。

缺陷草稿字段：

| 字段 | 来源 |
| --- | --- |
| title | 用例标题 + 失败摘要 |
| description | AI 汇总实际结果和复现步骤 |
| reproduce_steps | 手工填写或 AI 从步骤生成 |
| expected_result | 用例预期结果 |
| actual_result | 执行实际结果 |
| environment | 执行环境快照 |
| attachments | 图片、视频、日志、截图 |
| severity | 用户选择或 AI 建议 |

## 20. 执行 Hook、环境初始化与自动日志采集

### 20.1 设计目标

自动化用例应只关注测试断言和业务步骤，公共能力由框架统一提供：

- 环境初始化。
- 设备状态恢复。
- 串口日志采集。
- ADB logcat 采集。
- 截图和 UI XML 采集。
- 日志上传和报告链接生成。

### 20.2 执行生命周期与 Try-Except-Finally 机制

为保证硬件和环境资源的安全释放，执行器引擎底层使用 `try...except...finally` 包裹执行生命周期：

```text
try:
    pre_run
    try:
        pre_case
        case_run
    except (AssertionError, DeviceError, Exception) as e:
        handle_and_classify_error(e)
    finally:
        post_case (自动采集日志、恢复设备状态)
finally:
    post_run (释放全局资源、生成报告)
```

| Hook | 说明 |
| --- | --- |
| pre_run | 执行批次开始前，准备代码、环境、设备池、工具实例 |
| pre_case | 单用例开始前，初始化设备、清理状态、开始日志采集 |
| case_run | 执行 Python 用例 |
| post_case | 单用例结束后，停止采集、拉取日志、上传附件、回写链接 |
| post_run | 执行批次结束后，生成报告、AI 解读、释放资源 |

### 20.3 Android 初始化模块

Android 初始化按配置执行，默认提供以下动作：

| 动作 | 示例 |
| --- | --- |
| wait_device | `adb wait-for-device` |
| unlock_screen | 解锁屏幕，关闭锁屏干扰 |
| go_home | `adb shell input keyevent KEYCODE_HOME` |
| force_stop_app | `adb shell am force-stop <package>` |
| clear_app_data | `adb shell pm clear <package>` |
| install_apk | `adb install -r <apk>` |
| clear_logcat | `adb logcat -c` |
| set_screen_awake | 设置屏幕常亮 |
| collect_props | 采集 `getprop`、电量、网络、存储等环境快照 |

这些动作由框架配置控制，不要求每个 Python 自动化用例重复实现。

### 20.4 自动日志采集模块

日志采集策略可按项目、计划、用例配置。

| 日志来源 | 采集方式 |
| --- | --- |
| adb_logcat | pre_case 开始，post_case 停止并上传 |
| serial_log | 串口后台采集，按用例时间窗口切片 |
| app_screenshot | 失败时自动截图 |
| ui_hierarchy | 失败时 dump UI XML |
| device_state | 失败时采集电量、网络、进程、磁盘等状态 |
| tool_output | 功耗、继电器、日志分析工具输出 |
| pytest_log | Python 执行 stdout/stderr 和 pytest junit/json |

### 20.5 云端存储策略

推荐优先支持 S3-compatible 存储，例如 MinIO。原因：

- 私有化部署友好。
- SDK 成熟。
- 权限和生命周期策略清晰。
- 可生成带过期时间的下载链接。

可扩展存储：

| 存储 | 适用场景 |
| --- | --- |
| MinIO | 私有化部署首选 |
| AWS S3/火山 TOS/阿里 OSS/腾讯 COS | 企业已有对象存储 |
| 飞书云文档/云盘 | 团队协作和报告分发 |
| 企业 NAS | 内网实验室大日志归档 |
| 本地文件系统 | 开发或单机部署 |

### 20.6 报告链接规则

自动化报告中每个失败用例必须展示：

- 失败摘要。
- 错误堆栈。
- pytest 日志链接。
- 串口日志链接。
- ADB logcat 链接。
- 截图或 UI XML 链接。
- 环境快照链接。

报告不内嵌大日志，只展示摘要和链接，避免报告体积过大。

## 21. 版本迭代与 AI 质量分析模块

### 21.1 核心概念

| 概念 | 说明 |
| --- | --- |
| ProjectVersion | 项目版本，如 App 1.3.0、固件 2.1.5、发布版本 R2026.05 |
| Iteration | 迭代/Sprint/里程碑，可以来自 Jira、飞书项目或手工创建 |
| QualityScope | 质量分析范围，包含项目、版本、迭代、时间窗口、需求集合 |
| QualityGate | 质量门禁，如 P0 通过率、严重缺陷数、未测需求数 |

### 21.2 外部项目映射

| 外部来源 | 映射到平台 |
| --- | --- |
| Jira FixVersion | ProjectVersion |
| Jira Sprint | Iteration |
| Jira Epic/Component | 模块、需求分组或迭代范围 |
| 飞书项目版本字段 | ProjectVersion |
| 飞书项目迭代/里程碑 | Iteration |
| 飞书多维表格字段 | 可配置映射到版本、迭代、模块 |

### 21.3 自动生成版本执行结果

```text
选择项目 + 版本/迭代
  -> 汇总关联需求
  -> 汇总覆盖用例
  -> 汇总测试计划和执行批次
  -> 汇总缺陷和阻塞项
  -> 生成执行结果统计
  -> AI 分析风险和发布建议
  -> 输出版本/迭代报告
```

### 21.4 AI 风险分析维度

| 维度 | 分析内容 |
| --- | --- |
| 需求风险 | 未覆盖需求、高优需求未测、需求变更频繁 |
| 用例风险 | P0/P1 用例失败、用例未评审、自动化覆盖低 |
| 执行风险 | 通过率下降、重复失败、环境异常、中断较多 |
| 缺陷风险 | critical/major 未关闭、缺陷重开、集中模块异常 |
| 设备风险 | 特定设备/固件/硬件批次失败率高 |
| 进度风险 | 迭代临近结束但执行不足、报告缺失 |
| 发布风险 | 质量门禁未达标，AI 给出 go/no-go/conditional-go 建议 |

### 21.5 质量门禁示例

```json
{
  "p0_pass_rate_min": 0.98,
  "critical_open_max": 0,
  "major_open_max": 3,
  "requirement_coverage_min": 0.95,
  "automation_pass_rate_min": 0.9,
  "allow_interrupted_runs": false
}
```
