# 迭代路线与交付计划

## 1. 建设策略

结合文章建议，建设顺序应避免一开始铺得过大，优先完成最能体现价值的闭环：

```text
需求导入 -> AI 生成用例 -> 人工采纳/启用 -> 执行 -> 报告 -> 通知
```

然后再扩展 App UI、嵌入式设备、串口、AI 归因、缺陷同步和更复杂的项目管理能力。

## 2. 阶段规划

### 阶段 0：调研与技术验证，1-2 周

目标：验证关键技术可行性。

交付物：

- 本文档集。
- 技术选型确认。
- 本地模型或 OpenAI-compatible API 调通。
- 配置中心原型确认，包括模型、邮件、通知和工具配置。
- AI 助手自然语言意图识别 Demo。
- uiautomator2 连接 Android 设备 Demo。
- ADB Shell 执行 Demo。
- pyserial 串口读写 Demo。
- 飞书/Jira API 连通性验证。

验收标准：

- 能从一段需求文本生成结构化用例 JSON。
- 能连接一台 Android 设备并截图。
- 能执行一条 ADB Shell 命令并返回结果。
- 能读取一次串口日志或完成串口 Mock。

### 阶段 1：MVP 主链路，3-5 周

目标：打通平台主链路。

范围：

- 登录、用户、角色、项目管理。
- 配置中心基础能力：权限、大模型、邮件、通知、需求字段映射。
- 需求手工导入和文档导入。
- 大模型配置和 Prompt 模板。
- AI 需求分析和候选用例生成，并明确 manual/automated 与自动化可行性。
- 用例采纳、启用、版本管理和 Python 自动化绑定。
- 手工执行体验：实际结果、复现步骤、图片/日志上传、缺陷链接。
- 手动执行、loop 配置、中断处理、基础执行报告、执行日志管理、公共环境初始化和失败日志自动上传。
- AI 助手支持一句话生成用例和查询项目状态。
- 飞书/企业微信/邮件任选一种通知。

不做：

- 复杂缺陷同步。
- 大规模设备池。
- AI 自动修复脚本。
- 完整 RAG 知识库。

验收标准：

- 一个项目可以完成需求到报告的完整闭环。
- AI 生成用例必须可人工采纳和编辑。
- 用例修改必须保留版本。
- 报告可以发送到通知渠道。

### 阶段 2：智能硬件执行能力，4-6 周

目标：支持嵌入式和 App UI 自动化执行。

范围：

- Device Agent。
- 工具箱/Skill 注册中心初版。
- Android 设备发现和设备锁。
- ADB Shell、logcat、截图、安装卸载。
- 串口命令执行和日志断言。
- 功耗采集工具和继电器控制工具接入。
- uiautomator2 App UI 执行器。
- 执行附件归档。
- 失败日志 AI 解读初版。

验收标准：

- 支持至少 2 台设备并发管理。
- 执行任务不会抢占同一设备。
- App UI 用例失败时保存截图和 UI XML。
- 串口日志可以按关键字/正则断言。

### 阶段 3：企业集成与评审流程，3-5 周

目标：接入真实研发协作流程。

范围：

- Jira 需求同步。
- 飞书项目/多维表格需求同步。
- 字段映射配置。
- 用例评审流程。
- 缺陷创建和外部缺陷关联。
- 日报、周报自动生成。
- Jira/飞书版本、迭代、里程碑同步和映射。
- 版本/迭代执行结果自动汇总。

验收标准：

- 能从 Jira 或飞书同步需求并生成用例。
- 用例可以按需求批量发起评审。
- 执行失败可创建或关联缺陷。
- AI 助手支持自然语言发起已启用用例集合执行。
- 每日/每周报告自动推送。

### 阶段 4：AI 增强与质量运营，长期迭代

目标：提升 AI 生成质量和质量决策能力。

范围：

- RAG 知识库：历史用例、历史缺陷、产品规范。
- AI 用例去重和覆盖率分析。
- AI 失败归因和聚类。
- UI 定位失败辅助修复。
- 自动化稳定性分析。
- 质量看板和趋势预测。

验收标准：

- AI 生成用例采纳率可统计。
- 失败原因可自动聚类。
- 报告可以输出质量风险、版本/迭代风险和建议动作。
- 平台能展示项目质量趋势。

## 3. 里程碑

| 里程碑 | 时间建议 | 关键产出 |
| --- | --- | --- |
| M0 | 第 1 周 | 技术验证和详细设计确认 |
| M1 | 第 4-6 周 | MVP 可演示版本，含配置中心基础能力和 AI 助手用例生成 |
| M2 | 第 8-12 周 | 支持 ADB、串口、uiautomator2、功耗、继电器工具的执行版本 |
| M3 | 第 12-16 周 | 接入 Jira/飞书、版本迭代映射、评审流程和 AI 助手自然语言执行 |
| M4 | 第 16 周后 | AI 质量运营和高级分析 |

## 4. 团队分工建议

最低配置：

| 角色 | 人数 | 职责 |
| --- | --- | --- |
| 测试开发/后端 | 1 | 后端、执行器、AI 接入、设备 Agent |
| 前端 | 1 | 管理台、用例、报告、配置页面 |
| 测试专家 | 1 | 用例模板、评审规则、智能硬件场景 |
| 项目负责人 | 1 | 需求优先级、推进试点、验收 |

如果只有 1 人负责，建议：

- 优先复用开源后台模板和 UI 组件库。
- 先做模块化单体，不做微服务。
- 先接一个模型、一个通知、一个需求来源。
- 先支持一条硬件链路，不追求所有设备类型。

## 5. 风险清单

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| AI 生成质量不稳定 | 用例不可用，团队不信任 | 强制人工评审，建立 Prompt 模板和反馈闭环 |
| 公司网络无法访问云模型 | AI 能力不可用 | 优先支持本地模型和 OpenAI-compatible 内网模型 |
| Jira/飞书字段差异大 | 同步失败或数据混乱 | 字段映射可配置，保留 raw_payload |
| 设备连接不稳定 | 自动化执行失败率高 | 引入 Agent 心跳、设备锁、重试和健康检查 |
| 串口/ADB 命令有安全风险 | 误操作设备 | 命令黑名单/白名单，危险命令二次确认 |
| 一期范围过大 | 延期、不可演示 | 坚持 MVP 闭环，非核心功能后置 |
| 历史用例迁移困难 | 推广阻力大 | 支持 Excel 导入，先迁移核心用例 |
| 报告只堆数据无结论 | 管理层价值感弱 | 引入 AI 解读，输出风险和建议动作 |

## 7. 开发建议

### 7.1 后端优先级

1. 数据模型和 Alembic 迁移。
2. 用户、项目、权限基础 API。
3. 模型配置和 AI 调用封装。
4. 需求导入和 AI 用例生成。
5. 用例采纳、版本、启用。
6. 执行任务和报告。
7. ADB/uiautomator2/串口执行器。
8. Jira/飞书/通知集成。

### 7.2 前端优先级

1. 登录和基础布局。
2. 项目与配置页面。
3. 需求列表和 AI 分析页面。
4. 候选用例确认页面。
5. 用例库和版本对比页面。
6. 执行中心和报告页面。
7. 设备和集成配置页面。

### 7.3 测试策略

- 单元测试覆盖模型调用适配器、字段映射、状态机、权限校验。
- 集成测试覆盖需求导入到用例生成。
- 使用 Mock 模型保证 AI 相关测试稳定。
- 使用 Fake Agent 模拟设备执行。
- 真实设备测试作为 nightly 或专项测试。

## 8. 关键指标

| 指标 | 说明 |
| --- | --- |
| AI 用例采纳率 | 采纳用例数 / AI 候选用例数 |
| 需求覆盖率 | 已关联用例需求数 / 总需求数 |
| 自动化率 | 自动化用例数 / 可执行用例数 |
| 执行通过率 | passed / total |
| 失败聚类率 | 被 AI 聚类的失败数 / 失败总数 |
| 报告准时率 | 按计划生成报告数 / 应生成报告数 |
| 设备在线率 | 在线时长 / 总时长 |
| 平均执行耗时 | 执行批次或用例平均耗时 |

## 9. 明天可运行版本范围冻结

为了确保明天能够交付可运行版本，当前阶段临时冻结为“最小可运行闭环”，以 `13-明天最小可运行方案与开工待办.md` 为准。

### 9.1 明天必须交付

- 登录和默认管理员账号。
- 项目创建和项目详情。
- 需求文本导入。
- Mock AI 或 OpenAI-compatible AI 生成候选用例。
- 用例采纳、编辑、启用。
- 手动创建执行批次。
- 手工录入或模拟执行结果。
- 基础报告和 AI Mock 解读。
- README 启动说明。

### 9.2 明天暂不交付

- Jira/飞书真实同步。
- ADB、串口、功耗仪、继电器真实接入。
- 分布式 Agent 调度。
- uiautomator2 真机执行。
- CI/CD 阻塞式集成。
- 版本/迭代质量门禁。
- 跨项目效能大盘。
- 复杂项目权限和评审流。

### 9.3 实施原则

- 能 Mock 的先 Mock，保证主流程完整。
- 能配置的先留配置入口，真实集成后续补。
- 能本地跑起来优先，Docker Compose 次优先。
- 数据结构先满足主链路，复杂表后续逐步迁移。
