# Token 消耗记录

## 1. 重要说明

当前对话环境未直接暴露真实 API token 计费数据，因此本文档不能保证记录的是平台实际消耗 token。后续若接入的大模型 API 返回 `usage` 字段，例如 `prompt_tokens`、`completion_tokens`、`total_tokens`，应以 API 返回值为准。

本文档用于两类记录：

- 项目开发过程中的“对话级估算 token”记录。
- 平台运行后每次 AI 调用的“模型 API 实际 token”记录。

## 2. 记录口径

### 2.1 对话级估算

适用于当前开发协作过程，字段如下：

| 字段 | 说明 |
| --- | --- |
| 日期 | 对话发生日期 |
| 轮次 | 第几次用户需求或开发交互 |
| 输入摘要 | 用户输入和上下文摘要 |
| 输出摘要 | 助手输出或生成文件摘要 |
| 估算输入 token | 根据文本长度估算，不作为计费依据 |
| 估算输出 token | 根据生成内容估算，不作为计费依据 |
| 估算合计 token | 输入 + 输出 |
| 累计估算 token | 项目累计估算 |
| 备注 | 说明限制或异常 |

### 2.2 API 实际 token

适用于平台后续真实调用大模型，建议落库到 `ai_tasks.token_usage`：

```json
{
  "prompt_tokens": 1234,
  "completion_tokens": 800,
  "total_tokens": 2034,
  "provider": "openai_compatible",
  "model": "qwen2.5-32b-instruct"
}
```

## 4. 后续维护方式

每次重要对话或开发阶段结束后追加一行记录：

```markdown
| YYYY-MM-DD | N | 输入摘要 | 输出摘要 | 输入估算 | 输出估算 | 合计 | 累计 | 备注 |
```

如果后续开发平台本身，需要在 AI 调用封装层记录真实 usage：

```python
def normalize_usage(response):
    usage = getattr(response, "usage", None) or {}
    return {
        "prompt_tokens": usage.get("prompt_tokens"),
        "completion_tokens": usage.get("completion_tokens"),
        "total_tokens": usage.get("total_tokens"),
    }
```

## 5. 平台内 token 统计建议

平台上线后建议提供以下看板：

| 维度 | 指标 |
| --- | --- |
| 项目 | 每项目每日 token 消耗、成本估算 |
| 用户 | 用户触发 AI 任务次数和 token 消耗 |
| 场景 | 需求分析、用例生成、报告解读、失败归因消耗对比 |
| 模型 | 不同模型调用量、失败率、平均耗时、平均 token |
| 质量 | token 消耗与用例采纳率、报告有效反馈的关系 |

## 6. 成本控制策略

- 对需求文本进行摘要后再生成用例，避免长文本反复送模。
- 使用 Prompt 模板版本化，减少无效输出。
- 对相同需求版本和 Prompt 版本做缓存。
- 低风险场景使用小模型，高复杂度场景使用大模型。
- 对日报/周报先结构化汇总，再让 AI 解读汇总数据。
- 记录 AI 输出采纳率，用数据淘汰低质量 Prompt。
