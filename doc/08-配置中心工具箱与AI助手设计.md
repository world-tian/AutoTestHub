# 配置中心、工具箱/Skill 与 AI 助手设计

## 1. 设计背景

在智能硬件测试平台中，配置项会非常多。如果没有统一配置中心，权限、大模型、邮件、通知、Jira、飞书、ADB、串口、功耗仪、继电器、执行超时、命令安全等配置会散落在不同模块，后期维护成本极高。

同时，智能硬件测试存在大量可复用能力，例如功耗采集、继电器控制、串口命令、ADB 操作、日志分析、数据生成、夹具控制。如果每个用例都重复实现，会导致脚本碎片化、不可维护、不可审计。

因此新增三类核心能力：

- 配置中心：统一管理平台所有可变参数。
- 工具箱/Skill：将可复用测试能力标准化封装。
- AI 助手：以自然语言作为显眼入口，调用需求、用例、执行、报告和工具箱能力。

## 2. 配置中心

### 2.1 配置分层

```text
系统级配置
  -> 项目级配置
  -> 测试计划级配置
  -> 用例级配置
  -> 运行时覆盖配置
```

优先级规则：

```text
运行时覆盖 > 用例级 > 计划级 > 项目级 > 系统级
```

### 2.2 配置分类

| 分类 | 典型配置 | 优先级 |
| --- | --- | --- |
| 权限配置 | 账号创建、邀请、角色、权限点、项目成员、审批规则 | P0 |
| 大模型配置 | Provider、base_url、model_name、api_key、参数、场景绑定 | P0 |
| Prompt 配置 | 需求分析、用例生成、报告解读、失败归因模板 | P0 |
| 邮件配置 | SMTP、发件人、收件组、邮件模板、测试发送 | P0 |
| 通知配置 | 飞书、企业微信、Webhook、事件订阅、消息模板 | P0 |
| 需求配置 | Jira、飞书、多维表格、字段映射、同步周期 | P0 |
| 执行配置 | 超时、并发、重试、loop 循环、Agent 策略、中断策略、报告策略、日志策略 | P0 |
| 命令安全 | ADB/Shell/串口黑白名单、危险动作确认 | P0 |
| 工具配置 | 功耗仪、继电器、串口服务器、夹具、实验室资源 | P1 |
| AI 助手配置 | 启用意图、可调用工具、确认策略、上下文范围 | P1 |

### 2.3 配置能力要求

- 支持新增、编辑、启用、停用、测试连接。
- 支持敏感值加密存储，例如模型 Key、SMTP 密码、机器人 Webhook。
- 支持配置版本管理和回滚。
- 支持配置变更审计。
- 支持 JSON Schema 校验配置结构。
- 支持配置导入导出。
- 支持执行时生成配置快照，保证报告可追溯。

### 2.4 配置页面建议

| 页面 | 核心功能 |
| --- | --- |
| 权限配置 | 账号策略、邀请策略、系统角色、项目角色、权限点、审批规则 |
| 大模型配置 | 模型 Provider、参数、场景绑定、连通性测试 |
| Prompt 配置 | 模板编辑、变量说明、版本、输出 Schema |
| 邮件配置 | SMTP、模板、收件组、测试发送 |
| 通知配置 | 飞书、企业微信、Webhook、事件订阅 |
| 需求源配置 | Jira/飞书字段映射、同步策略 |
| 执行配置 | 超时、并发、重试、loop 循环、Agent 选择、中断报告、日志保留策略 |
| 工具配置 | 功耗仪、继电器、串口服务器、夹具实例 |
| 安全配置 | 命令黑白名单、脱敏、危险动作确认 |
| AI 助手配置 | 意图开关、可调用工具、确认策略 |

## 3. 工具箱/Skill

### 3.1 目标

工具箱是平台的“能力复用层”，负责把底层硬件、仪器、脚本、日志能力封装成标准工具，供测试用例、测试计划、AI 助手和人工调试页面调用。

### 3.2 工具抽象

```text
ToolDefinition：工具定义，描述工具能做什么
ToolInstance：工具实例，描述某个具体设备或服务
ToolAction：工具动作，描述可调用的方法
ToolInvocation：调用记录，保存输入输出和证据
ToolArtifact：工具产物，保存日志、CSV、曲线、截图等附件
```

### 3.3 工具类型

| 工具 | 动作示例 | 使用场景 |
| --- | --- | --- |
| PowerMeterTool | start/stop/measure/export_curve | 待机功耗、峰值电流、续航验证 |
| RelayTool | on/off/power_cycle/pulse | 断电重启、上下电循环、模拟按键 |
| LogAnalyzerTool | search/extract/cluster/summarize | 串口、logcat、服务日志分析 |
| AdbUtilityTool | shell/logcat/screenshot/install/uninstall | Android 设备操作复用 |
| SerialUtilityTool | send/read_until/extract | 嵌入式串口命令和日志分析 |
| DataFactoryTool | generate_account/device_name/random_payload | 测试数据生成 |
| NetworkTool | ping/check_port/set_proxy/weak_network | 网络连通性和弱网测试扩展 |
| FixtureTool | press/rotate/touch/sensor_mock | 夹具、机械臂、传感器模拟扩展 |

### 3.4 工具接入形态

| 形态 | 说明 |
| --- | --- |
| Python Plugin | 适合平台内置 SDK、功耗仪 SDK、继电器 SDK |
| HTTP Tool | 适合调用已有实验室服务或第三方设备平台 |
| Shell Tool | 适合命令行工具，必须受黑白名单限制 |
| Agent Tool | 适合由测试机本地执行 USB、串口、ADB、仪器操作 |

### 3.5 用例调用工具示例

```json
{
  "case_type": "mixed",
  "steps": [
    {
      "type": "tool",
      "tool_name": "RelayTool",
      "action": "power_cycle",
      "params": {"channel": 1, "off_seconds": 5}
    },
    {
      "type": "tool",
      "tool_name": "PowerMeterTool",
      "action": "measure",
      "params": {"duration_seconds": 60, "sample_rate": 10},
      "assertions": [
        {"type": "less_than", "field": "avg_current_ma", "value": 120}
      ]
    }
  ]
}
```

### 3.6 工具安全

- 每个工具声明风险等级：low/medium/high/critical。
- 高风险工具必须配置二次确认。
- 工具实例需要按项目授权。
- 工具调用必须保存完整审计记录。
- Shell 类工具默认不开启，必须配置白名单。
- 继电器、刷机、重启、擦除数据等动作默认属于高风险。

## 4. AI 助手

### 4.1 产品形态

AI 助手应是平台一级入口，建议采用以下形态：

- 首页顶部全局输入框：“想做什么测试？”
- 右下角悬浮助手，支持跨页面上下文。
- 项目页内助手，默认带当前项目上下文。
- 需求、用例、执行报告详情页内助手，默认带当前对象上下文。

### 4.2 核心能力

| 能力 | 示例 |
| --- | --- |
| 生成用例 | “根据这条需求生成 P0/P1 测试用例，并标注哪些适合自动化” |
| 查询状态 | “智能门锁项目今天执行情况怎么样？” |
| 发起执行 | “执行 WiFi 模块冒烟用例，使用 lab-phone-01 和固件 1.2.3” |
| 调用工具 | “采集设备待机 5 分钟功耗，判断平均电流是否小于 100mA” |
| 分析失败 | “分析今天失败最多的 10 个用例，按可能原因聚类” |
| 生成报告 | “生成本周回归报告，输出风险和建议，发到飞书群” |

### 4.3 可控 Agent 流程

```text
用户自然语言
  -> 意图识别
  -> 项目/需求/用例/设备上下文补全
  -> 参数缺失时追问
  -> 权限校验
  -> 生成执行计划
  -> 风险评估
  -> 用户确认
  -> 调用用例、执行器或工具箱
  -> 生成结果摘要和报告
```

### 4.4 执行计划示例

```json
{
  "intent": "run_cases",
  "project": "智能门锁",
  "suite": "低功耗回归",
  "environment": "lab-a",
  "devices": ["lock-dev-01", "lab-phone-01"],
  "tools": [
    {"name": "PowerMeterTool", "action": "measure"},
    {"name": "RelayTool", "action": "power_cycle"}
  ],
  "risk_level": "high",
  "need_confirmation": true,
  "report": {
    "ai_interpretation": true,
    "notify": ["feishu_quality_group"]
  }
}
```

### 4.5 安全边界

- AI 助手不能越权访问项目数据。
- AI 助手不能启用未评审或未启用的正式执行用例，除非用户有对应权限并明确确认。
- AI 助手不能调用未注册工具。
- AI 助手不能执行任意 Shell 命令。
- AI 助手不能未经确认执行高风险工具。
- AI 助手所有动作必须落库，包括意图、计划、确认人、工具调用、执行结果和 token 消耗。

## 5. 与测试用例的关系

AI 助手和工具箱不替代用例管理，而是增强用例执行：

```text
AI 助手负责理解自然语言和编排动作
工具箱负责提供可复用能力
执行器负责运行测试步骤
用例库负责沉淀可评审、可版本化、可复用的测试资产
```

正确使用方式：

- 临时探索：AI 助手可以直接调用低风险工具查询状态。
- 正式测试：AI 助手应选择已启用用例或创建测试计划执行。
- 新场景沉淀：AI 助手生成候选用例，经人工采纳和评审后进入用例库。

## 6. MVP 建议

第一阶段建议先实现：

- 配置中心：模型、邮件、飞书/企微通知、权限、需求字段映射。
- 工具箱：ADB 工具、串口工具、日志分析工具的标准化封装。
- AI 助手：一句话生成用例、查询项目执行状态。

第二阶段再实现：

- 功耗仪工具。
- 继电器工具。
- AI 助手自然语言发起已启用用例集合执行。
- 高风险动作确认和审计闭环。

第三阶段增强：

- AI 助手自动选择设备和工具。
- AI 失败聚类和根因建议。
- 工具市场/插件市场。
- 多 Agent 分布式实验室资源调度。

## 7. 项目级配置与协同边界

项目是协同和权限隔离的核心边界。除系统管理员外，用户必须成为项目成员或所属团队被授权，才能访问项目内需求、用例、执行、报告和配置。

项目级配置建议包括：

| 配置 | 说明 |
| --- | --- |
| 成员权限 | 项目成员、团队授权、项目角色、自定义权限 |
| 流程规则 | 用例评审、AI 采纳、执行审批、高风险动作确认 |
| AI 策略 | 默认模型、Prompt、AI 助手可用意图、token 限额 |
| 需求源 | Jira、飞书、多维表格、字段映射、同步周期 |
| 用例规则 | 字段模板、优先级、标签、版本策略、启用规则 |
| 执行策略 | 默认环境、设备池、并发、重试、超时、定时 |
| 工具授权 | 功耗仪、继电器、串口服务器、Agent、夹具资源 |
| 通知报告 | 通知通道、日报、周报、报告模板、订阅人 |
| 安全审计 | 命令黑白名单、脱敏、审计保留周期 |

协同建议：

- 项目 Owner 负责项目创建、归档、成员和最高权限。
- 项目管理员负责项目配置和资源授权。
- 测试负责人负责用例评审、测试计划和质量结论。
- 研发、产品、项目经理以协作角色参与评审、缺陷分析和报告确认。

## 8. 执行 loop、中断和日志配置

### 8.1 配置继承

执行相关配置支持全局默认和单用例覆盖。

```text
运行时覆盖 > 用例级配置 > 测试计划配置 > 项目配置 > 系统默认配置
```

如果单个用例没有配置 `execution_config`，则自动复用上层有效配置。

### 8.2 全局执行配置示例

```json
{
  "execution": {
    "timeout_seconds": 1800,
    "retry_count": 0,
    "loop": {
      "enabled": false,
      "count": 1,
      "interval_seconds": 0,
      "stop_on_first_failure": false,
      "collect_logs_each_loop": true
    },
    "interrupt": {
      "generate_local_report": true,
      "notify_on_interrupted": false,
      "release_device_lock": true,
      "collect_remaining_logs": true
    },
    "logs": {
      "level": "info",
      "retain_days": 30,
      "split_upload_mb": 20,
      "mask_sensitive_data": true
    }
  }
}
```

### 8.3 单用例覆盖示例

```json
{
  "test_case_id": "case-id",
  "execution_config": {
    "loop": {
      "enabled": true,
      "count": 50,
      "interval_seconds": 3,
      "stop_on_first_failure": true
    },
    "timeout_seconds": 7200
  }
}
```

### 8.4 中断报告策略

- `generate_local_report`：默认 true，中断后必须生成本地报告。
- `notify_on_interrupted`：默认 false，中断报告不发送到飞书、企业微信、邮件。
- `collect_remaining_logs`：默认 true，中断时尽量采集设备和 Agent 当前日志。
- `release_device_lock`：默认 true，中断后释放设备锁和工具锁。

## 9. 公共初始化与日志采集配置

公共初始化和日志采集属于框架能力，不应该散落在每个 Python 自动化用例中。

配置项：

| 配置 | 说明 |
| --- | --- |
| environment_init_profile | Android/嵌入式设备初始化动作集合 |
| log_collection_policy | logcat、串口、截图、UI XML、设备状态采集策略 |
| artifact_storage | MinIO/S3/飞书云盘/NAS/本地文件系统 |
| failure_artifact_policy | 失败时上传哪些日志和附件 |
| report_link_policy | 报告中展示签名链接、内网链接或飞书链接 |

配置继承：

```text
用例级 > 计划级 > 项目级 > 系统级
```

默认建议：

- 通过用例只保留摘要日志。
- 失败、错误、中断用例上传完整日志。
- 报告中展示日志链接，不内嵌大日志。
- 串口日志按用例执行时间窗口切片。
- ADB logcat 在 pre_case 清空并开始采集，post_case 停止并上传。
