# 账号体系、项目权限与协同设计

## 1. 设计目标

平台的核心价值不仅是执行自动化测试，还要统筹软件测试全流程，方便测试、研发、产品、项目经理在同一个项目空间中协同工作。

因此账号和项目不应只是基础登录功能，而应承担以下职责：

- 账号体系：统一用户身份、登录、邀请、停用、审计。
- 项目空间：作为需求、用例、计划、执行、报告、缺陷、配置的隔离边界。
- 权限控制：控制谁能看、谁能改、谁能执行、谁能配置、谁能审批。
- 协同流程：支撑评审、评论、待办、通知、质量结论和项目交付。
- 项目配置：按项目管理模型、需求源、通知、工具、设备、执行策略和安全策略。

## 2. 账号创建方式

### 2.1 推荐支持方式

| 方式 | 说明 | MVP 建议 |
| --- | --- | --- |
| 管理员创建 | 系统管理员在后台创建账号，设置初始密码或发送激活邮件 | 必做 |
| 邀请加入 | 平台管理员或项目 Owner 邀请用户加入平台或某个项目 | 必做 |
| 自注册 | 用户自行注册，需配置是否开放、邮箱域名限制、管理员审核 | 默认关闭 |
| SSO/OIDC | 企业统一登录，平台不保存密码，只维护业务权限 | 二期 |
| LDAP/组织同步 | 同步企业组织、用户和团队 | 二期 |
| 飞书组织同步 | 从飞书组织架构同步用户，适合飞书项目深度集成 | 二期 |

### 2.2 MVP 推荐策略

MVP 阶段建议采用：

```text
系统管理员创建首个管理员
  -> 管理员创建用户或邀请用户
  -> 用户激活账号
  -> 加入项目并分配项目角色
```

原因：

- 自注册容易带来无归属用户和权限风险。
- 企业 SSO 初期接入成本较高。
- 管理员创建 + 邀请加入最适合快速落地。

### 2.3 账号状态

```text
invited -> active -> disabled -> locked -> archived
```

状态说明：

| 状态 | 说明 |
| --- | --- |
| invited | 已邀请但未激活，不能登录 |
| active | 正常账号，可登录 |
| disabled | 被停用，不能登录，历史记录保留 |
| locked | 安全锁定，例如连续登录失败 |
| archived | 归档账号，通常用于离职或长期停用 |

## 3. 权限模型

### 3.1 总体模型

建议采用：

```text
系统 RBAC + 项目 RBAC + 项目 ACL + 配置权限
```

含义：

| 模型 | 解决问题 |
| --- | --- |
| 系统 RBAC | 谁能管理全局用户、系统配置、全局模型、全局工具 |
| 项目 RBAC | 谁在某个项目里是什么角色 |
| 项目 ACL | 用户或团队是否能访问某个项目 |
| 配置权限 | 谁能改模型、通知、工具、执行、安全等配置 |

### 3.2 系统角色

| 系统角色 | 权限 |
| --- | --- |
| SuperAdmin | 平台最高权限，用户、项目、系统配置、审计全部可管 |
| SystemAdmin | 管理用户、团队、全局配置、全局集成 |
| PlatformOps | 管理工具模板、公共设备、通知模板、质量运营看板 |
| Auditor | 查看审计日志、配置变更、执行记录，无编辑权限 |
| NormalUser | 普通用户，只有被授权项目的权限 |

### 3.3 项目角色

| 项目角色 | 权限 |
| --- | --- |
| ProjectOwner | 项目最高权限，成员、角色、配置、归档、删除 |
| ProjectAdmin | 管理项目配置、成员、设备、工具、通知和执行策略 |
| TestLead | 管理用例评审、测试计划、执行、报告和质量结论 |
| Tester | 维护需求、用例、执行任务、分析结果 |
| TestDeveloper | 管理自动化脚本、执行器、工具箱、Agent 和设备 |
| Developer | 查看需求、用例、失败日志、报告，参与缺陷定位 |
| ProductManager | 查看需求覆盖、用例评审、报告和质量风险 |
| Viewer | 只读查看授权数据 |

### 3.4 权限点示例

| 资源 | 权限点 |
| --- | --- |
| project | read/update/archive/manage_members/manage_roles/manage_config |
| requirement | read/import/sync/update/delete/analyze |
| testcase | read/create/update/delete/adopt/enable/review/export |
| execution | read/create/run/stop/retry/delete/schedule |
| report | read/create/export/notify/ai_analyze |
| defect | read/create/update/sync |
| device | read/create/update/delete/lock/unlock/authorize |
| toolbox | read/create/update/delete/invoke/debug/authorize |
| config | read/update/test_connection/rollback |
| assistant | chat/generate_cases/run_cases/run_tools/analyze |
| audit | read/export |

## 4. 项目空间设计

### 4.1 项目是核心边界

项目下应包含：

- 成员与角色。
- 需求和需求源配置。
- AI 模型和 Prompt 配置。
- 用例库、用例版本、评审流程。
- 测试计划、用例套件、执行批次。
- 设备池、工具箱授权、执行 Agent。
- 报告、通知、日报、周报。
- 缺陷关联和外部缺陷同步。
- 项目审计日志。

### 4.2 项目创建流程

```text
选择项目模板
  -> 填写项目基础信息
  -> 选择所属团队
  -> 添加项目 Owner 和管理员
  -> 初始化项目角色
  -> 初始化项目配置
  -> 邀请成员
  -> 配置需求源、模型、通知、工具和设备
```

### 4.3 项目模板

项目模板用于快速初始化配置。

| 模板 | 适用场景 | 默认能力 |
| --- | --- | --- |
| App 测试项目 | 移动 App 功能和 UI 自动化 | uiautomator2、ADB、App UI 用例模板 |
| 嵌入式项目 | 固件、串口、硬件功能测试 | 串口、ADB、日志分析、设备版本字段 |
| 智能硬件项目 | App + 固件 + 硬件联动 | ADB、串口、功耗、继电器、设备池 |
| 接口测试项目 | 后端接口测试 | API 执行器、接口用例模板、环境变量 |
| 综合项目 | 端到端质量管理 | 全部模块按需启用 |

## 5. 项目级配置

### 5.1 配置分类

| 配置页 | 内容 |
| --- | --- |
| 基础信息 | 项目名称、Key、描述、所属团队、可见性、状态 |
| 成员权限 | 成员、团队授权、项目角色、自定义权限、邀请记录 |
| 流程规则 | 用例评审、AI 用例采纳、执行审批、缺陷创建策略 |
| AI 配置 | 默认模型、Prompt、AI 助手能力、token 限额 |
| 需求配置 | Jira、飞书、多维表格、字段映射、同步周期 |
| 用例配置 | 用例字段模板、标签规范、优先级、版本策略 |
| 执行配置 | 默认环境、并发、重试、超时、loop 循环、中断报告、日志保留、定时任务、CI Webhook |
| 设备工具 | 设备池、Agent、功耗仪、继电器、串口服务器、夹具 |
| 通知报告 | 飞书、企业微信、邮件、订阅人、日报、周报、模板 |
| 安全审计 | 命令黑白名单、危险动作确认、脱敏规则、审计保留 |

### 5.2 项目配置继承

```text
系统默认配置 -> 项目配置 -> 测试计划配置 -> 用例配置 -> 执行时覆盖
```

规则：

- 项目配置可覆盖系统默认配置。
- 测试计划可覆盖项目执行配置。
- 用例可覆盖部分执行器参数。
- 执行开始时保存配置快照，保证结果可复现。

### 5.3 谁能改项目配置

| 配置 | 可修改角色 |
| --- | --- |
| 基础信息 | ProjectOwner、ProjectAdmin |
| 成员权限 | ProjectOwner、ProjectAdmin |
| 流程规则 | ProjectOwner、ProjectAdmin、TestLead 部分可改 |
| AI 配置 | ProjectOwner、ProjectAdmin |
| 需求配置 | ProjectAdmin、TestLead |
| 用例配置 | ProjectAdmin、TestLead |
| 执行配置 | ProjectAdmin、TestLead、TestDeveloper |
| 设备工具 | ProjectAdmin、TestDeveloper |
| 通知报告 | ProjectAdmin、TestLead |
| 安全审计 | ProjectOwner、ProjectAdmin，部分需系统管理员 |

## 6. 协同工作台

### 6.1 页面结构

项目工作台建议作为进入项目后的主界面。

| 页面 | 说明 |
| --- | --- |
| 概览 | 质量指标、风险、需求覆盖、执行趋势、待办 |
| 需求 | 需求导入、同步、AI 分析、覆盖状态 |
| 用例 | 候选用例、正式用例、版本、评审、启用 |
| 计划 | 测试计划、套件、定时、执行策略 |
| 执行 | 执行批次、实时日志、失败分析、重跑 |
| 报告 | 执行报告、日报、周报、AI 解读、通知记录 |
| 缺陷 | 缺陷创建、关联、同步、状态跟踪 |
| 设备 | Android 设备、嵌入式设备、串口、Agent、设备锁 |
| 工具 | 功耗仪、继电器、日志分析、数据生成、工具调试 |
| 成员 | 成员、团队、角色、邀请、权限审计 |
| 配置 | 项目级所有配置入口 |

### 6.2 待办中心

待办来源：

- 项目邀请待接受。
- AI 候选用例待采纳。
- 用例待评审。
- 测试计划待执行。
- AI 执行计划待确认。
- 高风险工具调用待确认。
- 执行失败待分析。
- 缺陷待处理。
- 报告待确认或发布。

### 6.3 协同动作

| 动作 | 场景 |
| --- | --- |
| 评论 | 需求、用例、执行结果、报告、缺陷 |
| @成员 | 请求研发定位、产品确认、测试补充用例 |
| 发起评审 | 用例、测试计划、报告结论 |
| 指派待办 | 失败分析、缺陷处理、报告确认 |
| 订阅通知 | 关注某个计划、模块、需求、设备 |
| 权限申请 | 用户申请加入项目或申请执行/配置权限 |

## 7. 推荐 MVP 范围

第一阶段必须做：

- 管理员创建账号。
- 邀请用户加入项目。
- 用户登录和基础资料。
- 系统角色和项目角色。
- 项目成员管理。
- 项目级配置入口。
- 项目级模型、需求源、通知、执行配置。
- 项目工作台基础页面。
- 操作审计。

第一阶段可以暂缓：

- 自注册。
- SSO/OIDC。
- LDAP/飞书组织同步。
- 自定义复杂权限表达式。
- 权限申请审批流。
- 团队空间复杂层级。

## 8. 安全与审计

必须审计：

- 创建、禁用、锁定用户。
- 邀请成员、移除成员、修改项目角色。
- 修改项目配置。
- 修改模型、密钥、通知、工具、命令安全策略。
- AI 助手发起执行或调用工具。
- 高风险工具确认。
- 执行任务触发、停止、重跑。

审计记录至少包含：

- 操作人。
- 项目。
- 资源类型和资源 ID。
- 操作类型。
- 变更前后摘要。
- 时间、IP、User-Agent。
