# 版本/迭代质量分析与 AI 风险报告设计

## 1. 设计目标

平台需要支持从项目、版本、迭代、周期多个维度统筹质量情况，并通过 AI 分析风险问题。常见问题包括：

- 某个版本当前执行结果怎么样？
- 本迭代还有哪些高风险需求没有覆盖？
- 最近一周项目失败率升高的原因是什么？
- 是否建议当前版本发布？
- Jira 或飞书项目里的版本、迭代如何和平台测试数据关联？

因此需要建立统一的版本/迭代模型，并将需求、用例、执行、缺陷、报告挂到对应范围上。

## 2. 核心对象

| 对象 | 说明 |
| --- | --- |
| ProjectVersion | 项目版本，如 App 1.3.0、固件 2.1.5、发布版本 R2026.05 |
| ProjectIteration | 迭代、Sprint、里程碑或发布窗口 |
| QualityScope | 质量分析范围，包含项目、版本、迭代、时间窗口 |
| QualityReport | 版本报告、迭代报告、周期风险报告、发布报告 |
| QualityGate | 质量门禁规则，用于判断是否达到发布标准 |

## 3. 外部系统映射

### 3.1 Jira

| Jira 概念 | 平台映射 |
| --- | --- |
| Project | Project |
| FixVersion | ProjectVersion |
| Sprint | ProjectIteration |
| Epic | 模块、需求组或迭代范围 |
| Component | ProjectModule |
| Issue | Requirement 或 Defect |

### 3.2 飞书项目

| 飞书项目概念 | 平台映射 |
| --- | --- |
| 空间/项目 | Project |
| 工作项 | Requirement 或 Defect |
| 版本字段 | ProjectVersion |
| 迭代字段 | ProjectIteration |
| 里程碑字段 | ProjectIteration 或 Milestone |
| 模块字段 | ProjectModule |

### 3.3 灵活映射原则

不同团队在 Jira 或飞书里字段差异很大，因此必须支持字段映射配置：

- 哪个字段代表版本。
- 哪个字段代表迭代。
- 哪个字段代表模块。
- 哪个字段代表需求优先级。
- 哪个字段代表发布状态。

## 4. 数据关联

版本/迭代需要关联：

- 需求：本版本或本迭代包含哪些需求。
- 用例：这些需求覆盖了哪些用例。
- 计划：本版本有哪些测试计划。
- 执行：本版本执行了哪些批次。
- 缺陷：本版本新增、遗留、阻塞哪些缺陷。
- 报告：本版本生成了哪些报告。

关系示例：

```text
ProjectVersion 1.3.0
  -> Requirements
  -> TestCases
  -> TestPlans
  -> ExecutionRuns
  -> Defects
  -> QualityReport
```

## 5. 自动生成版本执行结果

用户选择项目和版本后，平台自动计算：

| 指标 | 说明 |
| --- | --- |
| 需求数 | 版本内需求总数 |
| 需求覆盖率 | 已关联用例需求数 / 总需求数 |
| 用例总数 | 版本关联用例总数 |
| 已执行用例数 | 在版本范围内已执行用例数 |
| 通过率 | passed / executed |
| 失败率 | failed / executed |
| 阻塞数 | blocked 用例和阻塞缺陷 |
| 自动化覆盖率 | 自动化用例数 / 总用例数 |
| P0/P1 通过率 | 高优用例通过情况 |
| 未关闭缺陷 | 按严重等级统计 |
| 失败聚类 | 按模块、设备、日志原因聚类 |

## 6. AI 风险分析

AI 输入：

- 版本/迭代范围。
- 需求覆盖数据。
- 用例执行数据。
- 失败日志摘要。
- 缺陷状态。
- 设备和环境异常。
- 历史趋势。

AI 输出：

```json
{
  "summary": "当前版本整体质量结论",
  "risk_level": "medium",
  "key_risks": [
    "P0 用例仍有 2 条失败",
    "固件 2.1.5 在 A 批次设备上失败率明显偏高"
  ],
  "failure_clusters": [
    {
      "name": "串口超时",
      "affected_cases": 8,
      "affected_devices": ["dev-01", "dev-03"],
      "possible_reason": "设备重启后串口未恢复"
    }
  ],
  "recommended_actions": [
    "优先修复 P0 失败用例关联缺陷",
    "补跑 A 批次设备稳定性用例"
  ],
  "release_recommendation": "conditional_go"
}
```

发布建议：

| 建议 | 含义 |
| --- | --- |
| go | 风险可接受，建议发布 |
| conditional_go | 有条件发布，需要确认遗留风险 |
| no_go | 不建议发布，存在阻塞风险 |
| insufficient_data | 数据不足，无法判断 |

## 7. 质量门禁

质量门禁可按项目、版本或迭代配置。

示例：

```json
{
  "p0_pass_rate_min": 0.98,
  "p1_pass_rate_min": 0.95,
  "critical_open_max": 0,
  "major_open_max": 3,
  "requirement_coverage_min": 0.95,
  "automation_pass_rate_min": 0.9,
  "blocked_case_max": 5
}
```

门禁结果应进入报告：

- 达标项。
- 未达标项。
- 数据来源。
- 负责人建议。
- AI 风险解释。

## 8. 页面建议

### 8.1 版本/迭代看板

- 版本列表。
- 迭代列表。
- 当前测试进度。
- 需求覆盖率。
- 用例通过率。
- 缺陷趋势。
- AI 风险结论。

### 8.2 版本报告页

- 范围说明。
- 执行概览。
- 高优用例结果。
- 失败聚类。
- 缺陷风险。
- 日志和证据链接。
- 质量门禁。
- AI 发布建议。

## 9. MVP 建议

第一阶段：

- 手工维护版本/迭代。
- 需求、用例、测试计划关联版本/迭代。
- 自动生成版本执行结果。
- AI 生成版本质量结论。

第二阶段：

- Jira FixVersion/Sprint 同步。
- 飞书项目版本/迭代字段同步。
- 质量门禁配置。
- 周期风险报告。

第三阶段：

- 发布前质量预测。
- 历史版本质量对比。
- 跨项目风险趋势分析。
