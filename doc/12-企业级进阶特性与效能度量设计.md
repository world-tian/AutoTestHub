# 企业级进阶特性与效能度量设计

## 1. 设计目标

为了让自动化测试平台真正达到企业级可用标准，不仅需要满足执行与报告的基本需求，还需要解决以下核心工程问题：

1. **测试环境与数据隔离**：一套代码如何在 Dev、Test、Staging、Prod 无缝切换？
2. **流水线阻塞集成**：CI/CD 如何等待测试结果并决定是否阻断发布？
3. **分布式资源调度**：海量硬件设备和执行机如何打标签、智能路由？
4. **测试资产 Single Source of Truth**：平台与 Git 仓库的代码如何保持同步不冲突？
5. **全局效能洞察**：管理层如何跨项目看自动化收益和质量大盘？

## 2. 测试数据与多环境管理 (Environment & Data-Driven)

### 2.1 多环境管理
项目内支持定义多个环境（Environment），如 `Dev`, `Test`, `UAT`, `Prod`。
每个环境包含一套独立的全局变量集合，例如：
- `api_gateway_url`
- `db_connection_string`
- `default_test_account`
- `device_pool_label`

**执行策略**：测试计划或手动执行时，必须选择一个目标环境。平台将对应的环境变量注入给执行器（例如作为 pytest 的环境变量或 config fixture）。

### 2.2 数据驱动 (Data-Driven Testing)
- **数据池 (Data Pool)**：支持在平台上维护 CSV/JSON 格式的测试数据集，供多条用例共享。
- **参数化配置**：用例级支持配置数据驱动参数，执行引擎会自动展开为多个子用例（对应 pytest 的 `@pytest.mark.parametrize`）。
- **AI 赋能**：在创建数据池时，允许通过 AI 自然语言生成符合特定规则的边界值、非法字符等测试数据。

## 3. CI/CD 深度集成与 Open API

### 3.1 阻塞式流水线集成 (Synchronous Execution)
传统的 Webhook 触发是“触发即忘”（Fire-and-Forget）。企业级流水线需要阻塞式集成。
- **提供 CLI 工具 / 插件**：提供 `autotesthub-cli` 或者 Jenkins/GitLab/飞书流水线专属插件。
- **轮询/长连接**：CI 插件调用 API 触发测试计划后，每隔 N 秒轮询平台状态。
- **状态码返回**：测试完成后，如果门禁通过，API 返回 `exit 0`；如果失败或门禁不通过，返回 `exit 1`，从而直接阻断 CI 流水线部署。

### 3.2 Open API 体系
平台必须暴露一套 RESTful Open API，用于外部系统整合。
- **鉴权机制**：基于项目级别的 API Token（支持过期时间与权限粒度控制）。
- **核心接口**：
  - `POST /api/v1/projects/{pid}/plans/{id}/trigger` (触发执行)
  - `GET /api/v1/projects/{pid}/plans/{id}/status` (查询状态)
  - `GET /api/v1/projects/{pid}/reports/{id}` (获取摘要报告)
  - `GET /api/v1/agents/status` (获取节点大盘)

## 4. 分布式执行节点 (Agent) 调度管理

智能硬件测试高度依赖特定的宿主机环境（如连接了特定的串口板、抓包工具或手机的 Mac/PC）。

### 4.1 节点标签化 (Node Labels)
每个 Agent 启动时，除了注册基础信息（IP、Hostname、OS），还需要注册自定义标签（Tags）。
例如：
- `os:mac-arm64`
- `capability:adb,serial,power-monitor`
- `location:lab-01`
- `dedicated_for:project_A`

### 4.2 智能路由调度 (Smart Routing)
- 测试用例或测试计划可以声明**依赖标签**（Required Tags）。
- 平台的调度中心（Dispatcher）维护一个任务队列。
- 当有空闲 Agent 的标签**完全匹配**任务的依赖标签时，才将任务下发给该 Agent。
- 支持并发度控制（Max Concurrent Runs）与心跳超时剔除。

## 5. 自动化代码同步规范 (云端管控与本地执行分离)

针对硬件自动化测试必须依赖本地物理设备（串口、ADB、夹具等）的客观限制，平台采用“云+端”分离架构。

### 5.0 AutoTestRunner (本地执行 SDK)
- 提供名为 `pytest-autotesthub` 的 Python 库（可托管在 GitHub 内部私有库）。
- 测试人员在本地开发调试自动化用例时，只需安装该插件：`pip install pytest-autotesthub`。
- 本地执行命令示例：`pytest tests/ --hub-project-id=123 --hub-token=abc`。
- 该 SDK 会自动捕获 pytest 的执行结果、错误堆栈、本地采集的日志，并通过 Open API 同步上报到 AutoTestHub 云端平台。

## 5.1 自动化代码同步规范 (Test as Code)

为避免平台 Web IDE 与本地 Git 仓库修改冲突，确立**单向同步**原则。

### 5.1 Git 作为 Single Source of Truth
- 自动化测试的 Python 脚本（`test_*.py`）必须托管在 Git 仓库中。
- 平台上的自动化用例代码为**只读**视图。

### 5.2 自动同步机制 (Local to Cloud)
自动化用例的流转核心在于**“代码在本地，元数据在云端”**。具体打通机制如下：

1. **本地装饰器标记 (Code Annotations)**
   在本地 Python 脚本中，使用 `AutoTestRunner` 提供的装饰器进行声明：
   ```python
   from autotest_runner import autotest

   @autotest(case_id="TC-001", req_id="REQ-102", title="验证蓝牙断开重连机制", priority="P0")
   def test_bluetooth_reconnect():
       # 测试逻辑...
       pass
   ```

2. **主动同步命令 (CLI Sync)**
   开发完成后，测试工程师在本地（或由 CI/CD 流水线）执行命令：
   `autotest-runner sync ./tests --hub-url=https://hub.local --token=xxx`
   - Runner 底层利用 Python `ast` 模块或 `pytest --collect-only` 机制，提取所有带有 `@autotest` 标记的函数元数据。
   - 通过 Open API 将元数据（用例名称、关联需求、文件路径等）推送到 AutoTestHub。
   - AutoTestHub 在云端数据库中创建或更新这些用例记录，状态标记为 `automated`，且界面显示为**只读**。

3. **手工用例流转**
   **手动测试用例**则完全在 AutoTestHub 平台上由测试或产品人员通过富文本、表格创建与维护，人工执行并流转状态。

## 6. 全局大盘与效能度量 (Metrics Dashboard)

为了支撑管理层的决策，平台提供跨项目的全局效能洞察看板。

### 6.1 质量度量 (Quality Metrics)
- **自动化覆盖率趋势**：(自动化用例数 / 总用例数) 的按周/月趋势。
- **用例有效性**：用例发现缺陷的比例，长期 Pass 但从未发现问题的用例可能需要重构或降级。
- **发布门禁拦截率**：平台拦截了多少次不合格的 CI/CD 发布。

### 6.2 AI 效能度量 (AI ROI)
- **AI 用例采纳率**：AI 生成的用例被人工接受并转为正式用例的比例。
- **AI 节省时长估算**：通过 AI 自动分析失败日志、生成数据，累计节省的人工工时。

### 6.3 资产与资源大盘 (Asset & Resource)
- **执行节点健康度**：各 Agent 的在线时长、离线频次、CPU/内存水位。
- **设备利用率**：自动化测试占用硬件设备的总时长与闲置率分析。
