# 明天最小可运行方案与开工待办

## 1. 目标

明天交付目标不是完成企业级终态平台，而是交付一套可以本地启动、可以演示核心闭环、后续可持续扩展的 MVP。

交付标准：

```text
一条命令启动
  -> 登录平台
  -> 创建项目
  -> 导入一条需求
  -> AI/Mock AI 生成候选用例
  -> 人工采纳并启用用例
  -> 创建一次执行
  -> 录入或模拟执行结果
  -> 生成报告
```

## 2. 明天必须完成的闭环

| 模块 | 明天范围 | 说明 |
| --- | --- | --- |
| 登录 | 必须 | 默认管理员账号，JWT 登录 |
| 项目 | 必须 | 项目创建、列表、详情 |
| 需求 | 必须 | 文本导入需求，不做真实 Jira/飞书同步 |
| AI 生成 | 必须 | 支持 Mock Provider，预留 OpenAI-compatible 配置 |
| 用例 | 必须 | 候选用例、采纳、编辑、启用、manual/automated 分类字段 |
| 执行 | 必须 | 手动创建执行批次，支持手工录入结果或模拟执行 |
| 报告 | 必须 | 执行结果汇总、失败项、AI Mock 解读 |
| 权限 | 最小 | 只做管理员/普通用户基础角色 |
| 配置 | 最小 | 模型 Provider 配置、系统基础配置 |

## 3. 明天明确不做

以下能力保留在设计文档中，但不进入明天交付：

- Jira/飞书真实同步。
- SSO/LDAP/飞书组织同步。
- 分布式 Agent 调度。
- ADB、串口、功耗仪、继电器真实接入。
- uiautomator2 真机 UI 执行。
- CI/CD 阻塞式集成。
- 版本/迭代质量门禁。
- 跨项目效能大盘。
- 复杂评审流和审批流。
- 云端日志存储。

## 4. 技术选型

明天建议采用最短路径：

| 层 | 选型 | 原因 |
| --- | --- | --- |
| 后端 | FastAPI | 开发快，接口文档自动生成，适合 AI/任务接口 |
| 数据库 | SQLite 起步，可切 PostgreSQL | 本地跑起来最快；模型设计兼容 PostgreSQL |
| ORM | SQLAlchemy 2.x | 后续迁移 PostgreSQL 方便 |
| 迁移 | Alembic | 保留正式迁移能力 |
| 前端 | React + Vite + TypeScript | 快速搭建管理台 |
| UI | Ant Design | 表格、表单、布局成熟 |
| 启动 | Docker Compose + 本地脚本 | 明天可一键启动 |
| AI | Mock Provider + OpenAI-compatible Provider | 无模型也能演示，有模型可配置接入 |

## 10. 明天验收路径

验收步骤：

```text
1. 执行 docker compose up 或本地启动脚本。
2. 使用默认账号 admin / AutoTestHub@123 登录。
3. 创建项目“智能硬件演示项目”。
4. 粘贴一段需求文本。
5. 点击“AI 生成用例”。
6. 查看生成的手工/自动化候选用例。
7. 采纳 2 条用例并启用。
8. 创建执行批次。
9. 录入一条 passed、一条 failed，并填写失败原因。
10. 查看执行报告和 AI Mock 解读。
```

通过标准：

- 服务能启动。
- 前后端能连通。
- 数据能落库。
- 用例能从需求生成。
- 用例能采纳和执行。
- 报告能展示统计和失败信息。
- README 写清楚启动步骤。

## 12. 风险与兜底

| 风险 | 兜底方案 |
| --- | --- |
| 前端时间不够 | 保留最小页面，复杂交互用表单和表格完成 |
| 数据库迁移耗时 | 先 SQLite + SQLAlchemy create_all，后续再 Alembic |
| AI 接入不稳定 | 默认 Mock Provider，真实模型作为可选配置 |
| Docker 调试耗时 | 先本地启动，Docker Compose 作为补充 |
| 权限模型复杂 | 明天只做管理员/普通用户，项目级权限后续补 |
| 报告样式耗时 | 先做 HTML/页面报告，不做 PDF |
